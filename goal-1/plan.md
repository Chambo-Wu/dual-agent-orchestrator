# Goal 1 Plan

## 需求概述
- 当前项目仍以双模型配置为主，需要梳理并支持“多模型配置”的启动方式。
- 需要新增多模型模式下的模型有效性探测/健康检测能力。
- 健康检测通过的模型才加入执行队伍；无效模型应被跳过，并具备可观测性。

## 当前上下文
- 本轮为 goal mode 初始化轮次，按要求在完成 `input.md`、`plan.md`、`tasks.md` 前不得修改代码。
- 用户当前打开的是 `Readme-CN.md`，推测项目中已存在双模型编排逻辑、配置读取逻辑、文档说明。
- 尚未读取代码实现细节；下一会话开始时需要先完整读取 goal 文件，再检查仓库内与模型配置、启动流程、健康检查相关的实现。

## 默认假设
- “多模型配置”指项目支持 2 个以上模型实例参与统一编排，而非仅主/副两个固定槽位。
- “启动多模型配置”至少包含：配置格式、启动入口、运行期装配、失败模型处理。
- “模型有效性（健康检测）”可以通过轻量 API 探测、配置合法性校验、必要的超时与错误分类来实现。
- 若现有架构支持 provider/model endpoint 抽象，则优先在现有抽象上扩展，而不是重写 orchestrator。
- 若存在 README/中文文档，需要同步更新使用方式。

## 风险
- 现有代码可能将“双模型”假设硬编码在配置结构、UI、调度器或日志输出中，改动面可能跨多个模块。
- 健康检测若设计不当，可能拖慢启动速度、误判正常模型、或在网络异常时阻塞主流程。
- 多模型并入执行队伍后，可能影响现有轮询、并发、回退、优先级、结果聚合逻辑。
- 若测试覆盖不足，可能出现配置兼容性回归，尤其是旧双模型配置无法启动。

## 执行方案
1. 读取并梳理当前项目中与模型配置、启动逻辑、执行队伍构建相关的代码和文档。
2. 确认双模型假设的落点：配置 schema、运行期数据结构、调度流程、UI/CLI 参数。
3. 设计兼容方案：在保留旧双模型配置可用的前提下，引入多模型配置格式。
4. 实现启动时的模型列表解析、标准化和执行队伍构建。
5. 实现健康检测：对候选模型逐个校验，记录成功/失败原因，只将有效模型纳入执行队伍。
6. 为异常场景补充日志、错误提示和降级路径，例如全部模型失效时明确失败。
7. 补充或更新测试，覆盖旧配置兼容、多模型启用、健康检测过滤、全失败等场景。
8. 更新文档，说明如何配置与启动多模型模式。
9. 在每三个 task 后执行大型全面检查-debug 循环并记录结论。
10. 全部完成后做最终全面 review、修缮与验证，再标记 goal 完成。

## 验证方式
- 静态验证：检查配置类型/schema、关键分支、日志与错误处理。
- 自动化验证：运行相关单元测试、集成测试、类型检查、构建命令。
- 手动验证：使用示例配置验证双模型兼容、多模型启动、部分模型失效过滤、全部模型失效报错。
- 文档验证：确保 README/中文文档步骤与真实启动方式一致。

## 回滚方案
- 将多模型支持设计为兼容扩展，保留原双模型配置路径。
- 若健康检测逻辑导致不稳定，可通过配置开关或受控默认值禁用新行为（具体是否已有配置能力，后续确认）。
- 代码修改保持小步提交，必要时可按提交粒度回退。
