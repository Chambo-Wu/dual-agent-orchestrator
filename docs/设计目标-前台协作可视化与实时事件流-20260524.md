# 前台协作可视化与实时事件流设计
> 文档状态：设计目标，请先阅读 README
>
> 本文主要用于说明前台协作可视化与实时事件流的目标形态、推荐协议与实施路线。
> 其中部分能力已经落地，但不应将全文视为当前实现清单；当前实际行为请以 [README](../README.md) 和 [Readme-CN](../Readme-CN.md) 为准。


- 日期：2026-05-24
- 目标：让前台用户看到 planner / executor / tool / system 之间的协作过程，而不是只有单一加载状态
- 范围：事件协议、实时流接口、前端展示模型、兼容性约束、分阶段实施方案

---

## 1. 设计目标

### 1.1 用户体验目标

前台在任务执行期间应能回答下面几个问题：

1. 系统当前在做什么
2. 是哪个模型或组件在做这件事
3. 为什么会进入下一步
4. 当前是否卡住、重试、降级或等待审批
5. 最终答案是如何逐步形成的

### 1.2 技术目标

1. 不破坏 OpenAI / Anthropic 兼容接口
2. 支持 Web、桌面端、Cherry Studio 类客户端之外的自定义前台
3. 支持实时订阅，也支持任务结束后回放全过程
4. 支持后续扩展到 team mode、多 worker、多 verifier
5. 默认展示结构化摘要，而不是裸露内部 prompt 或完整 chain-of-thought

### 1.3 非目标

1. 不把内部原始 prompt 直接展示给最终用户
2. 不把所有日志原样透传到前台
3. 不要求现有兼容客户端自动支持协作视图

---

## 2. 核心原则

### 2.1 双通道原则

前台必须区分两类数据流：

1. `answer stream`
   负责最终回答内容
   继续使用 `/v1/chat/completions`、`/v1/responses`、`/v1/messages`

2. `workflow stream`
   负责协作过程、步骤、工具、重试、降级、审批、artifact 更新
   使用独立控制面接口

这样做的原因是：

1. 标准模型协议关注“回答”
2. 协作事件流关注“过程”
3. 两者混在一起会破坏兼容客户端的类型假设

### 2.2 摘要优先原则

前台显示的不是完整思维链，而是结构化摘要：

1. `reasoning_summary`
2. `decision_summary`
3. `tool_summary`
4. `evidence_summary`
5. `status_transition`

### 2.3 统一事件原则

无论是单 task、team mode、research、code、file_ops，前台都只消费同一套事件 schema。

## 3. 用户界面模型

### 3.1 推荐布局

推荐三栏布局：

1. 左栏：Job 概览
   - goal
   - 当前状态
   - 当前 orchestrator step
   - 运行时能力
   - 审批状态

2. 中栏：协作时间线
   - planner 卡片
   - executor 卡片
   - tool 卡片
   - system 卡片
   - final 卡片

3. 右栏：产物与证据
   - artifact 列表
   - 当前搜索结果
   - 当前读取文件
   - 最终报告

### 3.2 卡片类型

建议前台至少实现这些卡片：

1. `planner`
   - 标题：规划下一步
   - 内容：本轮意图、下一步动作、是否进入执行

2. `executor`
   - 标题：执行任务
   - 内容：接收到的指令、允许的工具、执行摘要

3. `tool`
   - 标题：工具调用
   - 内容：工具名、参数摘要、成功或失败、产物

4. `system`
   - 标题：系统状态
   - 内容：重试、降级、审批、取消、恢复

5. `final`
   - 标题：最终结果
   - 内容：最终回答、验证状态、输出文件

### 3.3 用户视图层级

默认显示简洁模式：

1. 谁在做事
2. 做了什么
3. 结果如何

展开后显示详细模式：

1. 工具参数摘要
2. artifact 预览
3. 失败原因
4. 重试原因
5. 证据缺口说明

## 4. 事件协议设计

### 4.1 统一事件对象

```json
{
  "id": "evt_01jy...",
  "job_id": "job_xxx",
  "seq": 12,
  "time": "2026-05-24T13:30:01.123Z",
  "agent": "planner",
  "phase": "decision",
  "type": "planner.decision",
  "title": "规划下一步",
  "summary": "先搜索 DeepSeek V4 功能，再补充同代模型对比",
  "status": "need_executor",
  "step": 1,
  "task_run_id": "taskrun_xxx",
  "meta": {
    "allowed_tools": ["web_search"],
    "verdict": "not_applicable"
  }
}
```

### 4.2 字段定义

固定字段：

1. `id`
   事件唯一 ID

2. `job_id`
   所属 job

3. `seq`
   单 job 内严格递增序号，前台按它排序

4. `time`
   ISO8601 时间

5. `agent`
   可取：
   - `planner`
   - `executor`
   - `tool`
   - `verifier`
   - `synthesizer`
   - `system`

6. `phase`
   可取：
   - `start`
   - `reasoning`
   - `decision`
   - `result`
   - `retry`
   - `approval`
   - `final`

7. `type`
   稳定的机器可读事件类型

8. `title`
   前台直接显示的短标题

9. `summary`
   前台直接显示的摘要

10. `status`
    当前动作状态，如：
    - `running`
    - `success`
    - `partial_success`
    - `failed`
    - `blocked`
    - `completed`
    - `awaiting_approval`

11. `step`
    当前 orchestrator step 编号

12. `meta`
    扩展字段，不要求前台完全理解

### 4.3 标准事件类型

建议第一版定义以下稳定事件：

#### Job 级

1. `job.created`
2. `job.started`
3. `job.completed`
4. `job.failed`
5. `job.cancelled`
6. `job.retried`
7. `job.awaiting_approval`
8. `job.resumed`

#### Planner 级

1. `planner.start`
2. `planner.reasoning`
3. `planner.decision`
4. `planner.finalized`
5. `planner.replanned`

#### Executor 级

1. `executor.start`
2. `executor.reasoning`
3. `executor.result`
4. `executor.partial_success`
5. `executor.blocked`

#### Tool 级

1. `tool.start`
2. `tool.result`
3. `tool.failed`
4. `tool.blocked`

#### Artifact 级

1. `artifact.created`
2. `artifact.updated`
3. `artifact.selected`
4. `artifact.readback`

#### System 级

1. `system.retry`
2. `system.degrade`
3. `system.loop_detected`
4. `system.verification_passed`
5. `system.verification_failed`

### 4.4 当前 `workflow.*` 到新协议的映射

现有运行时已经能发出：

1. `workflow.step.start`
2. `workflow.planner.decision`
3. `workflow.executor.start`
4. `workflow.executor.result`
5. `workflow.tool.start`
6. `workflow.tool.result`

建议新增一个映射层，把它们转换成前台协议：

1. `workflow.step.start` -> `planner.start`
2. `workflow.planner.decision` -> `planner.decision`
3. `workflow.executor.start` -> `executor.start`
4. `workflow.executor.result` -> `executor.result`
5. `workflow.tool.start` -> `tool.start`
6. `workflow.tool.result` -> `tool.result`

不要把旧的 `workflow.*` 直接暴露给前台作为长期协议。

## 5. 控制面 API 设计

### 5.1 目标接口

新增一个专门的实时事件流接口：

1. `POST /v1/jobs`
   创建 job

2. `GET /v1/jobs/:id`
   获取 job 状态

3. `GET /v1/jobs/:id/events`
   拉取历史事件

4. `GET /v1/jobs/:id/stream`
   实时 SSE 订阅事件

### 5.2 `GET /v1/jobs/:id/events`

返回示例：

```json
{
  "job_id": "job_xxx",
  "count": 4,
  "events": [
    {
      "id": "evt_1",
      "seq": 1,
      "time": "2026-05-24T13:30:00.000Z",
      "agent": "system",
      "phase": "start",
      "type": "job.started",
      "title": "任务开始",
      "summary": "已开始执行任务",
      "status": "running",
      "step": 0,
      "meta": {}
    }
  ]
}
```

查询参数建议支持：

1. `since_seq`
2. `limit`
3. `agent`
4. `type`

### 5.3 `GET /v1/jobs/:id/stream`

SSE 事件示例：

```text
event: job.event
data: {"id":"evt_12","job_id":"job_xxx","seq":12,"agent":"planner","type":"planner.decision","title":"规划下一步","summary":"进入执行阶段","status":"need_executor","step":1,"meta":{"allowed_tools":["web_search"]}}

event: heartbeat
data: {"time":"2026-05-24T13:30:03.000Z"}
```

建议事件类型：

1. `job.event`
2. `job.snapshot`
3. `heartbeat`
4. `job.completed`
5. `job.failed`

### 5.4 为什么不用 `/v1/chat/completions` 承载协作事件

原因如下：

1. 标准客户端会把所有 `data:` 当成模型 chunk
2. 协作事件不符合 `choices/error` 的标准结构
3. 会再次出现兼容性错误

所以：

1. 标准模型接口只传模型回答
2. 控制面接口只传协作过程

## 6. 后端实现模型

### 6.1 事件总线

建议增加一个内存事件总线：

```ts
interface WorkflowUiEvent {
  id: string;
  jobId: string;
  seq: number;
  time: string;
  agent: "planner" | "executor" | "tool" | "system" | "verifier" | "synthesizer";
  phase: string;
  type: string;
  title: string;
  summary: string;
  status: string;
  step?: number;
  taskRunId?: string;
  meta: Record<string, unknown>;
}
```

组件建议：

1. `event-normalizer`
   把内部运行时事件转成 UI 事件

2. `job-event-bus`
   按 job 保存最近事件，并向 SSE 订阅者广播

3. `event-store`
   把事件持久化到 job record 或独立 jsonl

### 6.2 事件来源

事件来源分四类：

1. orchestrator `onEvent`
2. job 状态变更
3. artifact 持久化
4. approval / retry / resume 控制动作

### 6.3 事件归一化策略

每个原始事件必须经过归一化：

1. 设置稳定的 `type`
2. 补全 `agent`
3. 生成可读 `title`
4. 生成前台可直接显示的 `summary`
5. 清理敏感信息
6. 限制 `meta` 大小

例如：

原始事件：

```json
{
  "type": "workflow.tool.result",
  "step": 1,
  "data": {
    "tool": "web_search",
    "ok": true,
    "summary": "Found 10 results (legacy)"
  }
}
```

归一化后：

```json
{
  "agent": "tool",
  "phase": "result",
  "type": "tool.result",
  "title": "工具返回结果",
  "summary": "web_search 返回 10 条结果",
  "status": "success",
  "step": 1,
  "meta": {
    "tool": "web_search",
    "ok": true
  }
}
```

### 6.4 快照机制

前台刚连接 SSE 时，不应该只能等后续增量事件。建议先推一个快照：

```json
{
  "job_id": "job_xxx",
  "status": "running",
  "current_step": 2,
  "latest_planner_summary": "正在整理搜索结果",
  "latest_executor_summary": "已读取 2 个 artifact",
  "artifact_count": 5
}
```

这样刷新页面后也能立刻恢复 UI。

## 7. 前端状态模型

### 7.1 前端 store

前端建议维护：

```ts
interface JobViewState {
  jobId: string;
  status: string;
  currentStep: number;
  timeline: WorkflowUiEvent[];
  artifacts: ArtifactPreview[];
  latestAnswer: string;
  latestSummary: string;
  approvals: ApprovalView[];
}
```

### 7.2 前端渲染规则

1. 同一步骤中的多个 tool 事件按时间折叠为一个小组
2. `reasoning` 与 `decision` 可以合并显示
3. `retry`、`degrade`、`loop_detected` 使用高亮状态条
4. `artifact.created` 提供快速预览入口
5. `final` 卡片固定显示在时间线底部

### 7.3 推荐展示层级

默认只显示：

1. 标题
2. 摘要
3. 状态
4. 时间

点击展开才显示：

1. `meta`
2. 参数摘要
3. artifact 路径
4. 失败详情

## 8. 安全与隐私约束

### 8.1 默认不展示

以下内容默认不直接发给前台：

1. 完整 system prompt
2. 完整推理原文
3. API key
4. 绝对路径中的敏感目录
5. 原始 shell 命令全文
6. 大段网页原文

### 8.2 可展示摘要

前台可以展示：

1. 决策摘要
2. 工具名
3. 工具结果摘要
4. artifact 名称或相对路径
5. 失败原因摘要
6. 降级原因摘要

### 8.3 审计模式

后续可以增加角色权限：

1. `user`
   只能看摘要

2. `developer`
   能看更详细 meta

3. `admin`
   能看完整调试事件

## 9. 分阶段实施方案

### Milestone A

目标：先让前台能“看到过程”

实现内容：

1. 定义 `WorkflowUiEvent` schema
2. 实现内部 `workflow.*` 到 UI 事件的归一化层
3. 扩展 `GET /v1/jobs/:id/events`
4. 前端渲染静态时间线

验收标准：

1. 一个 job 完成后可完整回放
2. 能看到 planner / executor / tool 三类事件
3. 不影响现有兼容接口

### Milestone B

目标：支持实时协作展示

实现内容：

1. `GET /v1/jobs/:id/stream`
2. heartbeat
3. 初始 snapshot
4. 前端实时订阅

验收标准：

1. 用户无需刷新即可看到步骤推进
2. 能在执行中看到工具调用和阶段变化

### Milestone C

目标：让过程“可理解”

实现内容：

1. 更好的 `title` 和 `summary` 生成
2. tool group 折叠
3. degrade / retry / loop 高亮
4. artifact 侧栏联动

验收标准：

1. 非开发者能大致看懂系统在做什么
2. 用户能区分正常推进和异常卡住

### Milestone D

目标：面向多 agent 扩展

实现内容：

1. team mode 事件标准化
2. verifier / synthesizer 事件
3. 多 task 并发视图
4. approval 视图

验收标准：

1. team mode 下可区分不同 agent 的工作流
2. 多任务依赖关系可视化

## 10. 与当前代码库的对接建议

### 10.1 已有基础

当前项目已经具备：

1. `runTask(..., onEvent)`
2. `workflow.*` 运行时事件
3. `GET /v1/jobs/:id/events`
4. job / orchestrator step / artifact 持久化

这意味着第一版不需要推翻现有架构，只需要补：

1. 事件归一化
2. 更稳定的事件 schema
3. 实时订阅接口
4. 前端消费模型

### 10.2 需要新增的模块

建议新增：

1. `src/workflow-ui-events.ts`
   - schema
   - normalizer
   - title / summary formatter

2. `src/job-event-bus.ts`
   - 订阅
   - 广播
   - seq 分配

3. `GET /v1/jobs/:id/stream`
   - SSE 输出

4. `GET /v1/jobs/:id/events`
   - 返回标准化后的事件，而不是仅从 job record 重建粗粒度时间线

### 10.3 事件持久化策略

建议优先用：

1. `runtime/jobs/<job_id>.events.jsonl`

优点：

1. 追加写简单
2. 回放方便
3. 不污染主 job record
4. 后续可支持增量拉取

## 11. 推荐的第一版落地顺序

最实用的顺序是：

1. 先定义 `WorkflowUiEvent`
2. 把现有 `workflow.*` 归一化并持久化
3. 改造 `/v1/jobs/:id/events`
4. 再补 `/v1/jobs/:id/stream`
5. 最后做前端时间线

这样可以保证：

1. 后端协议先稳定
2. 前端不依赖日志格式
3. 标准兼容接口不再被污染

---

## 12. 结论

这套前台协作展示的关键，不是让标准模型接口吐出更多过程，而是把“回答协议”和“协作协议”分离。

推荐的最终形态是：

1. `/v1/chat/completions` 等接口继续只负责回答
2. `/v1/jobs/:id/events` 负责历史回放
3. `/v1/jobs/:id/stream` 负责实时协作过程
4. 前台基于统一事件 schema 渲染时间线、卡片流和 artifact 面板

这样既能满足“看到大模型之间的交流”，也能保持通用性、跨平台性和协议兼容性。
