## 1. 里程碑计划（来自本轮对话）

### Milestone 1（测试与可注入运行时）

目标：把系统从“可跑”提升到“可验证、可回归”。

- 建立分层测试骨架：`test/unit`、`test/integration`、`test/e2e-lite`
- 引入 fake planner / fake executor 注入点
- 增加编排核心集成测试（runTask / runTeam / API contract）
- 将关键回归断言从单文件回归脚本迁移到分层测试

### Milestone 2（team runtime 主路径）

目标：把 `runTeam()` 从外壳能力收敛为一等运行时。

- decomposition 正式 schema 校验
- decomposition 非法时受控 fallback（不再粗暴退化）
- 增加 `planOnly`（只规划不执行）
- 强化 shared memory 在上下游任务间的信息流
- CLI 增加 team plan-only 入口

### Milestone 3（服务化控制面）

目标：补齐工作流控制能力与可操作 API。

- `AbortSignal` 贯通 runtime
- 引入轻量 job-store（`runtime/jobs/` 持久化）
- 暴露 `GET /v1/jobs/:id`、`GET /v1/jobs/:id/artifacts` 等查询接口
- 新增 `POST /v1/jobs/:id/cancel`、`POST /v1/jobs/:id/retry`
- 将 cancel 从“仅写元数据”推进到“真实中断运行中 job”

## 2. 已实现任务清单（按里程碑）

## Milestone 1 已完成

1. 测试骨架与脚本
- 已新增并使用：`test/unit`、`test/integration`、`test/e2e-lite`
- `package.json` 已具备分层测试脚本（`test`、`test:unit`、`test:integration`、`test:e2e-lite`）

2. fake 注入层
- 新增：`src/runtime/deps.ts`
- planner/executor/team 关键执行路径支持注入替身依赖，便于场景化测试

3. 集成测试起步并扩展
- 已覆盖：`runTask`、`runTeam`、基础 API contract
- 已补关键行为：loop detect、artifact readback、protocol correction 等

4. 运行时修正
- 修复 team 初始 pending 任务派发时序问题
- 清理 team timeout 悬挂定时器，避免测试句柄泄漏

## Milestone 2 已完成

1. decomposition schema
- 新增：`src/team-schema.ts`
- 校验项包含：必填字段、重复 title、未知 agent、未知依赖、循环依赖、memoryScope/retry 字段约束

2. `runTeam()` 受控 fallback
- decomposition 非法时降级为单个受控任务，而非“所有 agent 同做同一目标”

3. `planOnly`
- `runTeam` 已支持 plan-only 行为
- CLI 已支持：
  - `team plan -- "<goal>"`
  - `team --plan-only "<goal>"`

4. shared memory 信息流
- 上游任务结果写入 shared memory
- 下游依赖任务可读取并拼接上下文
- 已有对应集成测试保障

## Milestone 3 已完成（当前阶段）

1. 取消语义贯通
- `AbortSignal` 已接入 provider/runtime/orchestrator/team 主路径
- 取消场景已有测试覆盖（pre-abort、mid-flight）

2. job-store 与查询 API
- 新增：`src/job-store.ts`
- 已支持：
  - `GET /v1/jobs`
  - `GET /v1/jobs/:id`
  - `GET /v1/jobs/:id/artifacts`

3. 控制面基础（cancel/retry）
- 已支持：
  - `POST /v1/jobs/:id/cancel`
  - `POST /v1/jobs/:id/retry`
- 重试关联元数据已落盘（`retryOf`、`retriedToJobId` 等）

4. cancel 实际中断（本轮完成）
- 新增：`src/job-runtime.ts`（active job session 注册表）
- `executeTaskGoal` 启动时注册 `AbortController`
- cancel 接口会触发运行中 job 的真实 abort
- 启动即落盘 job 占位记录，避免运行中 cancel 命中 404
- 新增“运行中取消会中断”的集成测试

## 5. 仍在 backlog 的后续项（建议下一阶段）

1. 将 cancel 从“单进程内 session 中断”扩展为“可跨进程/重启恢复的控制语义”
- 当前 `job-runtime` 是内存注册表，进程重启后 active session 丢失
- 可考虑引入持久化 job session/state 机（或外部 store）

2. retry 策略增强
- 增加可选 retry 参数（覆盖模型/策略）
- 增加 retry lineage 查询能力（链式追踪）

3. runtime contract 再收敛
- 统一 task/team 在状态与 payload 上的更多边界行为
- 增补异常路径 contract 测试（例如 cancellation + retry 竞态）

4. observability 深化
- 对 team runtime 增加更细粒度事件（assignment/retry/synthesis timeline）
- dashboard 增加 job 控制事件可视化
