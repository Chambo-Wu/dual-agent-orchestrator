# 【部分完成】Dual Agent Orchestrator 路线图与实施清单

- 日期：2026-05-26
- 文档状态：收口版基线
- 适用范围：当前 `dual-agent-orchestrator` 主仓库
- 最新补充：2026-05-27 新增 Phase 4 verification layer 二次收口日志，详见 [路线图日志-Phase4验证层缺口-20260527.md](./路线图日志-Phase4验证层缺口-20260527.md)

## 1. 当前判断

项目已经越过“原型能跑”和“基础功能补齐”阶段，当前更准确地处于“核心能力已完成，进入平台化收口、产品化打磨与统一化扩展”的阶段。

现状可概括为：

- `planner + executor` 主执行链路已稳定可运行
- `workflow_plan` 的解析、校验、执行、replan 已落地
- `/v1/jobs`、`/events`、`/stream`、`/timeline` 已具备正式控制面与可视化能力
- `approval`、`artifact`、`workflow DAG`、`superseded history`、`replay / resume` 已进入正式语义
- `system-first verifier`、输出校验、runtime diagnostics、`doctor` 已具备可用基础
- 集成测试、构建与恢复链路已有较好覆盖

因此，当前阶段不应再把重点放在零散补功能，而应聚焦以下四类收口目标：

1. 将已落地能力显式标记为已完成，减少文档与实现漂移
2. 把阶段 1-3 剩余事项压缩为少量明确的增强项，而不是继续保留为大范围待办
3. 把阶段验收标准从定性表述升级成可验证条件
4. 把主要新增工作重心切换到 team-mode 与统一多代理契约

## 2. 总体推进原则

### 2.0 统一术语

为避免实现、接口、前端和规划文档之间继续漂移，后续默认采用以下术语：

- `job`：通过 `/v1/jobs` 管理的顶层执行记录
- `workflow`：挂在 job 上的结构化执行计划；运行中可被 `replan`
- `task`：workflow 中的节点，例如 `write`、`search`、`fetch`、`verify`、`synthesize`
- `task run`：task 在 job 内的一次持久化执行记录
- `step`：planner / executor 的一步迭代，或事件中的阶段标记；不等同于 `task`
- `artifact`：由工具或 task 执行产出的可持久化结果
- `verifier`：system-first 的验证层，用来判断输出是否真实、有效、充分
- `retry`：对同一任务意图重新执行
- `resume`：通过控制面继续一个中断或 blocked 的 job
- `replan`：在失败或新信息出现后替换或调整 active workflow
- `replay`：通过 `/events` 或 `/stream` 结合 `since_seq` / `Last-Event-ID` 重放持久化事件

### 2.1 先稳内核，再扩能力

后续阶段应优先处理状态语义、恢复能力、验证机制和事件契约，再推进 team-mode 与更复杂的多代理能力。

### 2.2 先补系统判定，再补模型判断

凡是能通过系统规则判断的内容，不应依赖模型自述。例如：

- 文件是否存在
- 路径是否在允许范围
- JSON 是否符合 schema
- artifact 是否真实产生
- 审批是否已被处理

### 2.3 先统一契约，再优化表现层

前端 timeline、streaming UX、DAG 交互应建立在稳定 contract 上，而不是反向驱动运行时结构。

### 2.4 team-mode 应建立在统一运行内核上

`runTask` 与 `runTeam` 后续应共享同一套：

- planner / executor step 语义
- artifact 语义
- verifier 语义
- approval 语义
- resume / retry / replan 语义

## 3. 阶段路线图

## 阶段 1：稳定平台版

- 当前状态：已完成

### 3.1 目标

让系统从“核心功能已具备”进入“可稳定依赖”的状态。

### 3.2 核心成果

- 统一 `job / workflow / task / step / approval / artifact` 状态语义
- 完成 durable execution 的基础闭环
- 明确 planner / executor / verifier 输出契约
- 补齐 job 生命周期级别的集成测试

### 3.3 当前判断

该阶段定义的核心问题已基本收口，当前残留内容主要属于持续维护，而不是阶段性缺口：

- 状态语义已基本统一，但仍需防止后续文档和用户可见输出再次漂移
- 运行中断、审批恢复、服务重启恢复已有主链路，后续只需维持测试覆盖
- workflow/replan 历史与恢复一致性已有基础闭环，后续主要是边界增强
- job 生命周期自动化验证已具备 integration coverage，后续以增量补测为主

### 3.4 验收标准

- 服务重启后，未完成 job 可恢复
- 审批阻塞后，可恢复执行
- replan 不破坏已归档的 superseded 历史
- `/v1/jobs` 主链路有稳定的 integration coverage

## 阶段 2：可信验证版

- 当前状态：已完成

### 3.5 目标

让“完成”从模型口头声明变成系统可证明的结果。

### 3.6 核心成果

- 建立 `system-first verifier`
- 将 verifier 结果结构化并纳入完成判定
- 为常见 task type 建立可复用验证规则
- 区分“直接失败”“应重试”“应 replan”“应 blocked”

### 3.7 当前判断

该阶段目标已基本达成，后续工作更偏增强而非补缺：

- 当前完成判定已不再只依赖 planner `final`，并已具备文件写入、artifact、验证结果等系统判定
- verifier 已进入正式执行语义与时间线消费路径，后续可继续增强为更细粒度的可组合能力
- planner 已在方向上转向依赖 verifier / runtime summary，后续仍可继续收紧对原始工具输出的直接耦合

### 3.8 验收标准

- 最终写文件、结构化结果、artifact 生成三类目标均可系统验证
- verifier 结果可稳定进入 job response / events / timeline
- 假成功率显著降低

## 阶段 3：产品化与可观测版

- 当前状态：进行中

### 3.9 目标

让系统不仅能运行，而且易于使用、排障、运维和对外集成。

### 3.10 核心成果

- 事件协议稳定化
- 运行观测与诊断能力增强
- timeline / DAG / artifact / replan 视图继续完善
- `doctor` 和 runtime profile 具备更强诊断价值

### 3.11 当前判断

该阶段主链路已基本形成，但仍有少量明显的产品化收口项：

- `/events` 与 `/stream` 的 replay / resume 语义已经可用，但长期契约文档仍可进一步显式化
- `doctor` 与 runtime profile 已具备诊断价值，但“如何使用这些诊断信息”仍缺少统一操作文档
- timeline 已具备 DAG / replan 聚焦能力，但 artifact / verifier / edge 联动仍有进一步抛光空间
- 可观察性已经不再只是展示层问题，后续重点更偏统计视图、文档化和前端交互完善

### 3.12 验收标准

- 客户端可稳定消费事件流并支持断线续传
- 出现异常时，能快速定位是模型、工具、配置、权限还是网络层问题
- timeline 能更好呈现 replan、artifact、verifier 上下文

## 阶段 4：统一多代理扩展版

- 当前状态：进行中

### 3.13 目标

将项目从“已具备 workflow 的 dual-agent runtime”升级为“统一多代理协作平台”。

### 3.14 核心成果

- 完成 `team-mode control plane`
- 统一 `runTask` 与 `runTeam` 内核语义
- 扩展多角色调度与更细粒度 replan
- 为未来多 worker / verifier / synthesizer 扩展打底

### 3.15 当前判断

- team-mode 已具备正式控制面入口，`/v1/jobs` 可通过 `mode: "team"` 创建、查询、重试、恢复和消费事件流
- agent registry、per-agent model routing、agent tool policy、agent limits 已形成最小运行时基础
- team job 已共享统一 taskRun / artifact / verifier / workflow DAG 语义，但 approval、角色化 verifier/synthesizer 与 patch 级 replan 仍需继续收口
- 后续 patch 级 replan、多角色状态继承和前端动态 agent registry 是进一步平台化的关键

### 3.16 验收标准

- `mode: "team"` 成为正式可运行路径
- team 与 task 共享统一状态、artifact、approval、verifier 契约
- replan 支持更细粒度的 plan 变更和历史保留

## 4. 当前优先级

当前不再按“先做阶段 1，再做阶段 2”的方式推进，而应按收口现状执行：

1. 维持阶段 1、2 已完成项的契约稳定
2. 收口阶段 3 剩余产品化与可观测性增强项
3. 将主要增量工作正式转入阶段 4 推进

原因：

- 阶段 1、2 已不再是主要瓶颈，继续作为大块 backlog 保留会模糊真实优先级
- 阶段 3 还存在少量高价值收口项，适合在进入 team-mode 前做完
- team-mode 是下一阶段最有价值的增量，但应建立在已稳定的事件、验证、诊断和恢复契约之上

## 5. 分阶段实施清单

本节改为按当前状态收口：

- `已完成`：当前仓库已具备，后续以维持契约稳定为主
- `进行中`：当前已具备基础，但仍有明确增强或抛光空间
- `未开始`：尚未进入正式收口阶段，作为下一阶段主任务

## 5.1 阶段 1收口清单：稳定平台版

### 已完成

1. 状态语义已基本统一
   - `job / workflow / task / step / approval / artifact` 已形成稳定语义
   - `partial_success / blocked / awaiting_approval / superseded` 已具备可运行含义

2. job 持久化与恢复主链路已落地
   - 服务重启后的 recoverable job 处理已具备基础闭环
   - approval 阻塞与 resume 主链路已进入正式控制面

3. 主链路 integration tests 已建立
   - 已覆盖 job 创建、事件流、审批恢复、retry、恢复等核心路径

4. 恢复一致性已有基础规则
   - workflow replan 后的 active / superseded 区分已进入持久化与事件模型

### 进行中

1. artifact 元数据仍可继续增强
   - 可继续补齐 `source`、`trust_level`、`related_step`、`related_task` 等更细语义

2. 输出 contract 仍可进一步收紧
   - 可继续区分“运行判断字段”和“展示摘要字段”

3. 命名与用户可见文案仍需持续清理
   - 重点减少历史 Milestone 命名残留和旧术语漂移

## 5.2 阶段 2收口清单：可信验证版

### 已完成

1. verifier 已进入正式语义
   - verifier 结果已纳入完成判定，不再只看 planner `final`

2. system-first checks 已有主路径实现
   - 文件存在、路径合法、artifact 非空、write target 匹配等已具备实际检查能力

3. verifier 结果已进入事件与时间线消费面
   - job response / events / timeline 已能承载验证相关结果

4. 异常分类已具备基础语义
   - 工具失败、模型失败、校验失败、审批阻塞、环境失败等已具备归类方向

### 进行中

1. verifier schema 仍需继续正式化
   - 将完整 `VerificationResult` 和 checks 结构持久化到 taskRun / job record
   - summary 只作为展示字段，结构化 checks 作为事实来源

2. 证据质量检查仍可继续增强
   - `search / fetch` 的来源数量、可信来源匹配、artifact 可读性仍有提升空间

3. planner 对 verifier summary 的依赖还可继续收紧
   - 减少 planner 对原始工具结果的直接耦合

4. workflow `verify` 节点仍需增强
   - 支持 task 级 verifier 选择
   - 支持 verifier profile / constraints
   - 支持 check 级事件与 artifact/task 关联

5. `insufficient` 验证语义仍需落地
   - 区分证据不足与执行失败
   - 将 insufficient 接入 retry / replan / blocked 策略

### 未开始

1. 模型级 verifier 接入 workflow 中间节点
   - 最终 job verification 已具备模型 verifier 补充路径
   - workflow `verify` 节点尚未接入模型 verifier
   - 后续应保持 system-first，模型 verifier 仅作为补充判断

## 5.3 阶段 3收口清单：产品化与可观测版

### 已完成

1. `/events` 与 `/stream` 协议已具备正式 replay / resume 主链路
   - 已支持 snapshot + replay
   - 已支持 `since_seq`
   - 已支持 `Last-Event-ID`

2. runtime diagnostics 已具备基础诊断价值
   - runtime profile 已可暴露平台、工具、依赖与诊断摘要

3. `doctor` 已具备正式输出
   - 已覆盖配置、模型 readiness、task routing、proxy、可写性、search provider、建议动作等

4. timeline / DAG / replan 已具备基础产品化展示能力
   - 已支持 dependency graph、replan history 与 lane focus

### 进行中

1. timeline 交互仍可继续增强
   - lane focus 持久化到 URL
   - replan 前后 plan 对比
   - artifact / verifier / edge 联动

2. 事件摘要与异常文案仍可继续抛光
   - 更好的 `title` 和 `summary`
   - 更清晰的异常原因分类文案

3. 运行分析视图仍未正式落地
   - 失败类型统计
   - 常见阻塞点
   - 工具使用分布

4. 观测操作文档仍待沉淀
   - 明确如何读 timeline、events、runtime-profile、doctor 输出

### 未开始

1. 控制面更细粒度查询能力
   - 过滤、分页、按类型回放目前仍不是正式收口项

## 5.4 阶段 4收口清单：统一多代理扩展版

### 已完成

1. 明确 team-mode 最小交付范围
   - 支持 `mode: "team"`
   - 支持最小 team plan
   - 支持 team job 查询、事件流、完成判定

2. 统一 `runTask` 与 `runTeam` 内核语义
   - team 子任务通过 `runTask` 复用 executor 执行链路
   - team job 已共用 taskRun、artifact、verifier、workflow DAG 语义
   - agent tool policy 已下沉到 executor request / tool execution 强约束

3. 补 team-mode integration tests
   - 覆盖 team job 创建、执行、失败、汇总、恢复

4. 推进更细粒度模型路由
   - agent registry 已支持不同执行者绑定不同模型配置
   - `default_executor_agent` 已支持非法或缺失 assignee 的默认回退
   - `limits.max_concurrency` 已接入 team dispatcher，支持 per-agent 并发配额

5. 强化多分支并发与依赖可视化
   - team job 已持久化 workflow graph / DAG
   - DAG 已包含 assignee、depends_on、attempts、verified、status 等 taskRun 语义

### 进行中

1. 扩展角色与调度
   - `worker`
   - `verifier`
   - `synthesizer`
   - `planner_proxy`

2. 推进 patch 级 replan
   - 在 replacement workflow 之外支持局部更新

3. 继续统一 approval / verifier / synthesizer 的 team 语义
   - approval 已有控制面语义，但 team 子任务级审批仍需进一步接入
   - verifier 已作为 job 完成判定共享，后续可扩展为角色化 verifier

4. 前端与配置控制面继续产品化
   - 当前 agent registry 主要来自 `config.yml`
   - 后续可推进前端动态配置、运行中可见的 agent registry 与能力说明

### 未开始

1. 评估未来插件化或外部代理接入边界
   - 仅在统一运行契约稳定后推进

## 6. 推荐近期执行顺序

如果只看接下来最值得立即推进的内容，建议按以下顺序执行：

1. 先做 Phase 4 verification layer 二次收口的文档与契约基线
   - 结构化 verification result 持久化方案
   - task 级 verifier 选择方案
   - `insufficient` 状态语义方案
2. 完成阶段 3 剩余收口项
   - timeline 交互抛光
   - 运行分析视图
   - 观测操作文档
3. 把阶段 1、2 的“进行中”增强项压缩成少量契约级改动
   - artifact metadata
   - verifier schema 显式化
   - planner 对 verifier summary 的依赖收紧
4. 正式转入阶段 4 team-mode 推进
   - 先定义 team-mode 最小可交付 contract
   - 再补 team-mode integration tests
   - 最后推进 patch 级 replan 和多角色扩展

## 7. 版本化建议

如果后续需要对外沟通阶段目标，可采用以下版本表达：

- `V0.8`：稳定平台版
- `V0.9`：可信验证与产品化版
- `V1.0`：统一多代理扩展版

对应含义：

- `V0.8` 重点是稳定、恢复、测试、契约
- `V0.9` 重点是 verifier、events、timeline、diagnostics
- `V1.0` 重点是 team-mode、统一多代理语义、细粒度 replan

## 8. 结论

当前项目最合理的推进方式，不是继续把阶段 1-3 视为大面积待办，而是将其视为“已完成或接近完成的收口阶段”，并将主要增量工作正式转入阶段 4 推进。

建议按以下顺序收口：

1. 先把运行内核做稳
2. 再把完成判定做可信
3. 然后把产品化与可观测能力补齐
4. 最后把 team-mode 与多代理扩展建立在统一契约上

只要按这个顺序推进，项目会从“已经具备不少能力的多模型 workflow runtime”，逐步演进为“稳定、可信、可恢复、可扩展的多代理协作平台”。
