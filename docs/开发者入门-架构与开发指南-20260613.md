## 核心架构

```
HTTP 请求 → src/index.ts (144行，CLI/export 聚合)
              └── src/server/router.ts
                    ├── /v1/chat/completions → src/server/chat-routes.ts
                    ├── /v1/messages          → src/server/chat-routes.ts
                    ├── /v1/responses         → src/server/chat-routes.ts
                    ├── /v1/jobs/*            → src/server/job-routes.ts
                    ├── /v1/goals/*           → src/server/goal-routes.ts
                    ├── /v1/skills/*          → src/server/skill-evolution-routes.ts
                    └── /v1/skill-evolution/* → src/server/skill-evolution-routes.ts

执行引擎:
  src/execution-service.ts  — Chat/Responses/Messages 到任务执行的入口
  src/task-execution.ts     — Job/Task/Team 执行、恢复、审批与测试注入点
  src/orchestrator.ts      — Planner/Executor 主循环（110KB）
  src/workflow-runtime.ts  — Workflow Plan DAG 执行与 Replan（52KB）
  src/workflow-plan.ts     — Plan Schema 解析/校验
  src/team.ts              — Team mode 多 Agent 协同

路由层（src/server/）:
  src/server/router.ts                 — HTTP 路由表与 handler 分发
  src/server/shared.ts                 — 共享 HTTP 工具（getRuntimeConfig, jsonResponse 等）
  src/server/auth.ts                   — API Key 鉴权
  src/server/sse.ts                    — SSE 写入工具
  src/server/skill-evolution-routes.ts — Skill Evolution API（13 handler，621行）
  src/server/goal-routes.ts            — Goal CRUD（11 handler，452行）
  src/server/job-routes.ts             — Job CRUD + Stream（17 handler，1043行）
  src/server/chat-routes.ts            — Chat/Responses/Messages（3 handler，797行）

响应与消息:
  src/job-response.ts       — Job response / workflow event / workflow summary 聚合
  src/server-response.ts    — Health response / workflow payload
  src/chat-message-utils.ts — OpenAI / Anthropic / tool message normalization
  src/progress-updates.ts   — 流式进度聚合与人类可读更新
  src/api-types.ts          — OpenAI/Responses/Anthropic request shared types

Skill Evolution:
  src/skill-evolution-automation.ts — 自动 reflect/propose/audit/validate/accept 流水线
  src/skill-evolution-builders.ts   — reflection/proposal 构建桥接
  src/skill-evolution-store.ts      — proposal/reflection/audit/validation/decision 持久化与路径护栏

CLI 入口（src/cli/）:
  src/cli/entry.ts         — main、server、task/team/dao-run 执行器
  src/cli/doctor.ts        — 配置诊断（doctor report、config checks）

### orchestrator.ts（110KB）
Planner/Executor 主循环。关键函数：
- `runOrchestrator()` — 顶层编排入口
- `runPlannerStep()` — 单次 Planner 调用
- `runExecutorStep()` — 单次 Executor 调用

### workflow-runtime.ts（52KB）
Workflow Plan DAG 执行引擎：
- `runWorkflowPlan()` — 主入口
- `executeFallbackTask()` — 失败回退
- `maybeReplanWorkflow()` — Replan 触发

### timeline.ts（130KB）
HTML Timeline 渲染。模板与逻辑混杂。重构方向：抽离 HTML 模板。

## 开发流程

```powershell
# 构建
npm run build

# 类型检查
npm run typecheck

# 配置校验
npm run config:validate

# 启动服务
npm run serve          # 标准
npm run serve:restart:9898  # 重启（端口 9898）

# 运行 CLI 任务
node --enable-source-maps dist/index.js "你的任务描述"

# 测试
npm run test:unit           # 单元测试（226 个）
npm run test:integration    # 集成测试
npm run test:skill-evolution-v2  # Skill 进化定向测试
npm run test:e2e-lite       # 轻量 E2E
```

## 测试编写

- 单元测试：`test/unit/*.test.ts`，使用 `node:test`
- 集成测试：`test/integration/*.test.ts`
- 测试辅助：`test/helpers/fake-runtime.ts` 提供最小配置和 fake 运行时

## 配置

- 模板：`config/example.config.yml`
- 实际：`config/config.yml`（从模板复制）
- 密钥：`.env`

## 重要约定

1. Planner Circuit Breaker：Planer 连续失败 3 次后熔断 60 秒
2. 文件写入校验：`write_file` 必须确认磁盘落盘后才报告成功
3. Verifier 主链：Workflow 每个 task 可经过 verifier 检查
4. Skill 自进化默认保守：`auto_validate=false`, `auto_accept=false`
5. Skill Evolution 路径护栏：live target 必须在 workspace 内，candidate / rollback target 必须留在 snapshot root 内
6. 测试配置覆盖：`setConfigOverrideForTests` 会克隆配置后归一化，不能依赖调用方 config 被原地修改
