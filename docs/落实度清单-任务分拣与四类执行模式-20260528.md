## 2. 落实度分级标准

为避免“好像做了”和“真正可用”混在一起，这里使用 4 级定义：

- `L0 未落地`
  - 仅在讨论或文档里出现
- `L1 有骨架`
  - 已有文件或入口，但功能不完整
- `L2 可运行 v1`
  - 真实接入主链路，可以实际工作，但仍存在明显缺口
- `L3 基本完整`
  - 具备独立 runtime、控制面、观测、验证与回退逻辑

## 4. 逐项落实度清单

已落实：

  - `src/intent-router.ts`
- 已有 4 类 intent
  - `direct_answer`
  - `research`
  - `goal`
  - `coding`
- 已有 heuristic 路由
- 已有 planner fallback
- 已接入真实入口
  - CLI
  - chat/messages
  - job/task 主链

证据：

- `src/intent-router.ts`
- `src/index.ts`
- `test/unit/intent-router.test.ts`

未落实：

- 没有形成单独的 `dispatchIntentRoute(...)` 调度层

  - 尚未引入 workspace-aware signals

### 4.2 direct_answer 执行模式

当前状态：`L2 可运行 v1`

## 5. 当前差距图

最关键的现实差距不是“有没有 4 个名字”，而是“有没有 4 套真正独立、可验证、可观测的执行模式”。

当前情况：

- `direct_answer`
  - 有独立文件
  - 但还比较薄
- `coding`
  - 有独立文件
  - 但还不是独立状态机
- `research`
  - 能力强
  - 但不是独立 runtime
- `goal`
  - 文档最完整
  - 代码落地最少

因此真正的主攻方向应该是：

1. `goal mode` 从概念落成控制面
2. `coding mode` 从 route policy 增强升级为独立执行状态机
3. `research mode` 从“默认主链”显式化为独立 runtime

## 6. 逐项可执行推进计划

下面的计划按“最小风险、最大增量”排序。

---

## 阶段 A：补齐执行模式调度层

目标：

- 把现在散在 `index.ts` 的路由执行逻辑收敛成一个显式调度层

建议新增：

- `src/intent-dispatch.ts`

建议职责：

- 输入：
  - `IntentRouteResult`
  - `goal`
  - `model`
  - `context`
- 输出：
  - 调用哪个 runtime

建议步骤：

1. 抽出 `dispatchIntentRoute(...)`
2. 让 `index.ts` 只做：
   - 解析输入
   - 调用 router
   - 调用 dispatch
3. 保持现有行为不变

验收标准：

- CLI / chat / messages / jobs 路径行为不变
- 相关测试不回归

优先级：高

---

## 阶段 B：把 research 显式化

目标：

- 把当前 research 主链从“默认路径”升级成“显式执行模式”

建议新增：

- `src/research-runtime.ts`

建议职责：

- 组装 research route policy
- 统一 research planner instruction
- 统一 research evidence contract
- 统一 research 结果总结

建议步骤：

1. 新增 `resolveResearchRoutePolicy(...)`
2. 新增 `runResearchIntent(...)`
3. 在 dispatch 层中显式调用
4. 保持 `orchestrator.ts` 内核不大改

验收标准：

- research 路由不再依赖“else fallback”表达
- route / events / logs 里能明确看到 research runtime 被选中

优先级：中高

---

## 阶段 C：把 coding 做成真正四阶段 runtime

目标：

- 从“增强 code route”升级到“编码任务状态机”

建议新增：

- `src/coding-phases.ts`
- 或扩展 `src/coding-runtime.ts`

建议阶段：

1. `understand`
   - 只允许找文件、读代码、grep
2. `edit`
   - 允许 write/edit
3. `verify`
   - 允许 build/test/check
4. `summarize`
   - 输出变更、验证、剩余风险

建议步骤：

1. 给 coding runtime 增加 phase state
2. 每阶段限制工具和目标
3. 产出结构化 phase events
4. 新增 coding-specific verification

验收标准：

- coding 流里能看到阶段切换
- timeline / dashboard 可显示 coding phase
- 测试能覆盖：
  - 只读分析
  - 修改
  - 验证失败
  - 总结

优先级：高

---

## 阶段 D：正式实现 goal mode v1

目标：

- 把当前最欠缺但价值最大的 `goal` 真正落地

建议新增：

- `src/goal-types.ts`
- `src/goal-store.ts`
- `src/goal-planner.ts`
- `src/goal-runtime.ts`
- `src/goal-contract.ts`

建议落地顺序：

### D1 初始化

1. `POST /v1/goals`
2. 生成：
   - `goal.json`
   - `input.md`
   - `plan.md`
   - `tasks.md`

### D2 读取

1. `GET /v1/goals`
2. `GET /v1/goals/:id`

### D3 推进一步

1. `POST /v1/goals/:id/run-next`
2. 每次只推进 1 个 task

### D4 检查循环

1. 每 3 个 task 自动插入一次 `large_check`

### D5 收尾

1. `POST /v1/goals/:id/review`
2. final review
3. 标记 completed

验收标准：

- goal 能跨轮次持久化
- 可暂停、继续、重试
- 每轮只执行一个 task

优先级：最高

## 阶段 E：给四类执行模式补齐统一 contract

目标：

- 四类模式不只是“路由名字不同”，而是有统一可观测 contract

建议新增字段：

- `selected_runtime`
- `execution_mode`
- `phase`
- `phase_summary`

建议所有模式都具备：

- route reason
- selected runtime
- entry policy
- verification summary
- fallback reason

验收标准：

- `/health`
- events
- timeline
- dashboard

都能看到模式级别元数据

优先级：中

## 7. 推荐推进顺序

最推荐顺序如下：

1. 阶段 A
   - 抽出 dispatch 层
2. 阶段 B
   - 把 research 显式化
3. 阶段 C
   - 把 coding 做成四阶段 runtime
4. 阶段 D
   - 正式落 goal mode v1
5. 阶段 E
   - 给四类模式补齐统一 contract

原因：

- A/B/C 风险小，能迅速把“4 类模式”从概念变成结构清晰的代码
- D 体量最大，放在前面容易阻塞
- E 是收口工作，适合在前面几步之后统一做
