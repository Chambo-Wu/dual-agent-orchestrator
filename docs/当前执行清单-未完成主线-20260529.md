### A5-1. Large check 语义补强

- 状态：done
- 当前事实：自动规划任务时已有 `large_check` 插入能力；显式传入 `tasks` 时默认不隐式改写，但可通过 `insert_large_checks: true` 显式请求插入

  - 显式传入 tasks：默认不隐式改写用户任务列表；如需要自动插入，传 `insert_large_checks: true`
- 已完成验收：
  - 行为有 unit 或 integration/API contract 覆盖
  - docs 明确自动规划与显式 tasks 的差异
  - 新读者不会误以为所有 create goal 场景都会无条件插入 `large_check`

- 状态：done
- 目标：把“可改进 skill”的想法沉淀为可审计 proposal，而不是直接改 `skills/`
- 核心项：
  - `src/skill-evolution-store.ts`

  - candidate dir
  - proposal API
- 验收：
  - proposal 可创建、读取、列出、更新状态
  - proposal 落在 `runtime/skill-evolution/`
  - 不直接污染 builtin skills

### B3. Auditor gate

- 状态：done
- 目标：在 proposal 应用前做 deterministic 审核
- 核心项：
  - schema 校验
  - patch scope 校验
  - tool/runtime/install source 不升级
  - 无 secret / task 泄漏
- 验收：
  - 高风险 patch 被拦截
  - proposal 状态机清晰
  - 单测覆盖关键拒绝场景

### B4. Deployment validation

- 状态：done
- 目标：让 candidate skill 与 baseline 做真实对比，而不是只看 proposal 文本
- 核心项：
  - baseline run
  - candidate run
  - validation report
- 验收：
  - candidate 未改进或变差时失败
  - candidate 确实改善时通过

### B5. Limited automation

- 状态：done
- 目标：仅对白名单低风险 skill 开启有限自动化
- 建议白名单：
  - `find.official_sources`
  - `find.workspace_files`
- 验收：
  - 非白名单 skill 不自动 accept
  - 自动化开关可控、默认保守
