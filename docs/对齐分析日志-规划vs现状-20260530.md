## 三、关键风险项

### P0 — 阻塞 Demo

1. **Demo 回归稳定性**：large_check、jobs dashboard 分页、tool 边界测试应进入固定 smoke/contract 组合
2. **长耗时 integration test 隔离**：GoalMode/Skill evolution 测试需要继续减少真实执行路径泄漏

### P1 — 影响阶段 3 收口

4. **replan 前后 plan 对比**：用户无法直观理解 replan 做了什么变更
5. **运行分析视图缺失**：无法快速定位系统瓶颈（失败类型、阻塞点、工具使用分布）
6. **events/stream 查询能力不足**：Jobs dashboard 已分页，事件回放仍缺少按类型/状态/分页查询

### P2 — 影响阶段 4 启动

7. **team 模式角色单一**：仅 worker 角色，verifier/synthesizer/planner_proxy 均未实现
8. **patch 级 replan 未实现**：replan 只能全量重规划，成本高
9. **Skill 进化运营视图缺失**：proposal queue/history/rollback 无 UI，运维靠命令行

### P3 — 技术债

10. **index.ts 过大**（8351 行）：单文件承载过多路由和逻辑，维护成本高
11. **research 模式过薄**（50 行）：实际能力完全依赖 planner，无独立策略
12. **文档与代码不同步**：部分 docs 将已完成项标为 todo，部分 todo 实际已完成

## 五、结论与建议

### 当前项目定位

项目处于**阶段 3 收口 + 阶段 4 启动**的过渡期。核心编排能力扎实（8.2/10），v1 功能链完整，但产品化细节和扩展能力仍有明显缺口。

### 下一步优先级建议

1. **固定 smoke/contract 回归组合**（P0）— large_check、jobs dashboard 分页、tool 边界作为 Demo 前置检查
2. **补全阶段 3 产品化细节**（P1）— replan 对比、运行分析视图、events/stream 查询
3. **启动阶段 4 角色扩展**（P2）— 从 verifier 角色开始，逐步扩展 team 模式
4. **Skill 自进化 v2 按需推进**（P3）— 优先推进 Deployment Validation v2 和 Observability v2
5. **index.ts 拆分**（技术债）— 8351 行单文件是维护隐患

### 不在 v1 范围内的事项（文档已明确排除）

- 任意远程 git 拉取并执行 skill
- Skill 沙箱执行器
- 跨 skill memory 图谱
- 复杂 skill 版本冲突解决
- Skill 市场与签名体系
- 多 goal 并发调度器
- 自动 git rollback 执行
