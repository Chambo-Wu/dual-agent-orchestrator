# 【规划中】Skill 自进化 V2 规划入口

- 日期：2026-06-01
- 最近更新：2026-06-14
- 定位：承接已落地的 Skill 自进化 v1 主链，定义 V2 的产品化、治理与自动化放大边界。
- 当前结论：V1 已基本可用，V2 不应继续重复 outcome / reflection / proposal / audit / validation 的基础实现，而应围绕“质量、可信度、可运营、安全放大”推进。2026-06-14 已完成 `src/index.ts` 架构拆分收口与 Skill Evolution 路径、类型、测试边界加固；后续新增能力应优先落在专门模块，而不是回填 `src/index.ts`。

## 1. V1 基线

当前已具备：

- skill outcome summary 与 reflection record。
- proposal store、candidate materialization、audit / validate / accept / reject API。
- deterministic isolated manifest replay、candidate workflow materialization、manual runtime replay validation。
- `skill_evolution.runtime_replay_in_auto_pipeline` 默认关闭的自动 pipeline runtime replay opt-in。
- proposal detail、ops summary、dashboard / timeline / event replay 基础观测。
- rollback snapshot、limited automation、risk tier automation ceiling 与动态风险信号基础。
- `SKILL.md` 双结构 inventory / check 脚本与首批 builtin skill 迁移。
- Skill Evolution path helper 已统一 live target、candidate snapshot、rollback snapshot 的路径解析；live target 必须留在 workspace 内，snapshot target 必须留在 snapshot root 内。
- Auditor / validator / accept / replay 均已接入统一路径 helper，避免 proposal target 路径策略漂移。

因此，V2 的默认假设是：基础闭环存在，后续任务只处理增强和放大。

## 2. V2 目标

V2 目标不是让系统更快自动改 skill，而是让自动化有足够证据、边界和运营解释。

1. 提升 proposal 质量：从最小安全 patch 走向结构化 rewrite 与真实场景 fixtures。
2. 提升 validation 可信度：明确 runtime replay 的 readiness、acceptance contract 与 auto-accept 保守边界。
3. 提升运营可见性：让 proposal queue、accepted history、rollback guide、aging、stuck-state 成为日常控制面。
4. 提升 auditor 深度：加入结构职责、跨文件一致性、verification diff、risk-tier aware audit。
5. 提升风险治理：将动态风险、冷却窗口、失败簇和自动化 ceiling 做成可解释策略。

## 3. 推荐工作包

### V2-A：Proposal Generator v2

- 状态：partially_done
- 当前已完成：四类 reflection fixtures、candidate content 断言、段落职责约束、manifest verification 白名单保护。
- 下一步：
  - 扩展真实 builtin skill 场景 fixtures。
  - 增加跨 `SKILL.md` 与 `skill.json` 的一致性断言。
  - 将 proposal quality 分为 safe / useful / regression-risk 三档，供 auditor 和 ops summary 消费。
- 验收：
  - 每类 reflection 至少覆盖一个真实 skill 场景。
  - candidate patch 能说明修改目标、修改段落、预期收益和风险。

### V2-B：Deployment Validation v2

- 状态：partially_done
- 当前已完成：task-level replay payload、workflow materialization、manual `stage=executed` runtime replay、自动 pipeline opt-in 边界。
- 下一步：
  - 明确 auto-accept 保守边界：`eligible` 必须依赖 runtime replay、same-input comparison、execution evidence 与 audit evidence。
  - 固化 non-regression + improvement acceptance contract。
  - 为 runtime replay 失败建立可解释分类：setup_failed、candidate_failed、baseline_failed、inconclusive、regression。
- 验收：
  - validation report 能解释通过、拒绝、无法判断三类结果。
  - auto pipeline 即使 opt-in，也不能绕过 dynamic risk、auditor 和 readiness gate。

### V2-C：Ops View v2

- 状态：partially_done
- 当前已完成：proposal queue 摘要、accepted history、rollback guide、funnel / aging 首批指标、dashboard 消费 ops summary。
- 下一步：
  - 独立 proposal ops 页面或等价 dashboard 分区。
  - stuck-state 分类：waiting_audit、waiting_validation、blocked_by_risk、needs_manual_review、rollback_available。
  - 提供按 skill、risk tier、proposal status 的过滤与排序。
- 验收：
  - 不打开单个 job timeline，也能判断 proposal 队列健康度。
  - 操作人能看出每个 proposal 卡在哪里、下一步该做什么。

### V2-D：Auditor Gate v2

- 状态：done
- 当前已完成：risk-tier aware audit 首批规则、section responsibility checks、identity drift 检查、verification contract diff whitelist、cross-file semantic consistency、high-risk manual review reason、remediation hints、verification diff quality linkage、统一 Skill Evolution path helper 接入。
- 下一步：
  - 继续基于真实 proposal 历史扩充审计 fixture。
  - 将 remediation hints 用于 proposal generator 下一轮自动修正仍属后续增强。
- 验收：
  - 审计结果不只是 pass/fail，还能说明失败类别、证据和建议修复方向。

### V2-E：Automation Risk Tiering v2

- 状态：partially_done
- 当前已完成：近期失败信号、replay stability、dynamic ceiling、auto-audit / auto-validate / auto-accept 前置拦截、cooldown 信号。
- 下一步：
  - 细化失败簇与可配置窗口。
  - 建立风险恢复策略：冷却期结束后如何从 high 降回 medium / low。
  - 将 risk decision 与 ops view 绑定，避免自动化阻断变成黑盒。
- 验收：
  - 每次自动化降级都能在 API / dashboard 看到原因、窗口、恢复条件。

## 4. 推荐顺序

第一批：

1. Proposal Generator v2 真实 skill fixtures。
2. Deployment Validation v2 auto-accept 保守边界确认。
3. Ops View v2 stuck-state 数据模型。

第二批：

1. Auditor Gate v2 cross-file semantic consistency。
2. Automation Risk Tiering v2 失败簇与恢复策略。
3. README / docs 再次收口。

第三批：

1. 独立 Ops 页面体验。
2. 更高风险 skill 的半自动改进试点。
3. 全量回归与发布说明。

- 不默认开启 `skill_evolution.runtime_replay_in_auto_pipeline`。

## 7. V2 不变量

推进 V2 时保持以下约束：

- `accept` 永远是可审计动作，不能变成不可追踪的后台副作用。
- runtime replay 是放大自动化的证据来源之一，但不是绕过 auditor 的理由。
- dynamic risk 可以降低 automation ceiling，但不能被 proposal 自己提高。
- high-risk skill 默认需要人工门槛。
- 所有 V2 新字段应保持向后兼容，旧客户端读取 v1 response 不应失败。

## 8. 暂不纳入 V2 首轮

- 默认开启高风险 skill 自动 accept。
- 让 agent 直接改写 live skill 并立即生效。
- 大规模迁移未验证的外部 skill。
- 在缺少 runtime replay 证据时放宽 auditor 或 dynamic risk gate。
