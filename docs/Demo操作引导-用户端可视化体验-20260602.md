# Demo 操作引导：用户端可视化体验

- 日期：2026-06-02
- 面向对象：首次体验 Dual Agent Orchestrator 的用户、演示人员、产品评审人员
- 目标：用 Desktop UI、浏览器页面和少量可复制命令，直观看到当前系统已具备的配置、job / workflow / timeline / goal / skill evolution / health 能力。

## 1. Demo 核心看点

本 Demo 不要求用户理解内部实现，重点让用户看到：

| 看点 | 用户能看到什么 | 对应能力 |
|---|---|---|
| Desktop 配置 | 直接编辑模型路由、`config/config.yml` 与 `.env` 常用密钥 | 新手首次配置入口 |
| 健康检查 | `/health` 返回模型配置、executor 探测、goal / skill evolution 摘要 | 运行时诊断与多模型准入 |
| Jobs Dashboard | job 列表、状态、模式、时间、Timeline 链接 | 持久化 job 控制面 |
| Job Timeline | planning / execution / verification / final 等事件卡片、runtime analysis、stream replay | workflow 事件流与可视化 |
| Artifacts / 结果 | job 输出、工件列表、本地文件落盘 | 任务结果与文件交付校验 |
| Goals Dashboard | goal 列表、任务状态、run-next / review 等操作入口 | Goal Mode v1 控制面 |
| Skill Evolution Ops | proposal queue、风险、eligibility、next action、accepted history | Skill 自进化治理与运营视图 |

## 2. 启动准备

### 路线 1：推荐新手使用 Desktop UI

在仓库根目录执行：

```powershell
npm install
npm run build
npm run desktop
```

Desktop UI 会自动启动内置本地 API，默认地址仍是：

```text
http://127.0.0.1:9898
```

首次体验建议先进入左侧 `模型` 页：

1. 在 `本地配置` 面板确认路径：
   - `config/config.yml`
   - `.env`
2. 填写常用密钥：
   - `PLANNER_API_KEY`
   - `EXECUTOR_API_KEY`
   - `SEARCH_API_KEY`
   - `EXECUTOR_REMOTE_API_KEY`
3. 在 `模型配置` 面板确认 planner / executor 的 `API URL`、`模型 ID`、`API Key` 引用。
4. 在 `路由` 面板确认 Planner 与 Executor 的组合。
5. 点击 `保存并重启`。
6. 切到左侧 `健康` 页，查看模型健康、executor probe 与服务日志。

Desktop UI 当前可以直接写入：

- `config/config.yml`：模型、角色、路由、部分保留配置段
- `.env`：常用 API key

因此新手不需要先手动打开配置文件，也能完成最小可运行配置。

### 路线 2：开发者使用浏览器和命令行

在仓库根目录执行：

```powershell
npm install
npm run build
npm run config:validate
npm run serve
```

默认服务地址：

```text
http://127.0.0.1:9898
```

演示前先打开页面：

- Jobs Dashboard：`http://127.0.0.1:9898/jobs/dashboard`
- Goals Dashboard：`http://127.0.0.1:9898/goals/dashboard`
- Skill Evolution Ops：`http://127.0.0.1:9898/skill-evolution/ops`
- Health：`http://127.0.0.1:9898/health`

如果 `/health` 返回 degraded，先说明这是模型服务或 API key 未就绪，不代表 UI 控制面不可用。Demo 可以继续展示历史 jobs / goals / skill ops 页面，但实时执行任务需要模型配置可用。

## 3. 推荐 Demo 路线

### 场景 0：Desktop UI 首次配置与总览

打开：

```powershell
npm run desktop
```

引导用户观察左侧导航：

- `执行`：输入目标，选择路由、模式与审批策略，直接创建 job。
- `模型`：编辑模型、路由、`config/config.yml` 和 `.env` 常用密钥。
- `Jobs`：内嵌 Jobs Dashboard。
- `Goals`：内嵌 Goals Dashboard。
- `Skill Ops`：内嵌 Skill Evolution Ops。
- `健康`：查看 `/health`、team agents、服务日志与当前配置路径。

在 `模型` 页完成一次配置演示：

1. 修改 planner 或 executor 的 `API URL` / `模型 ID`。
2. 将 `API Key` 保持为 `env:PLANNER_API_KEY` 或 `env:EXECUTOR_API_KEY`。
3. 在 `本地配置` 面板填写对应 `.env` 密钥。
4. 点击 `保存并重启`。
5. 切到 `健康` 页，确认服务已重启，并查看健康 JSON。

讲解重点：

- Desktop UI 是新手入口，能避免手动编辑 YAML 和 `.env`。
- 配置仍然落到仓库内的 `config/config.yml` 与 `.env`，不是只存在桌面壳里。
- 如果健康页显示 degraded，用户可以直接看到服务日志和配置路径，便于修正。

### 场景 A：健康检查与运行时状态

Desktop UI 路线：

1. 点击左侧 `健康`。
2. 点击右上 `刷新` 或页面内 `刷新`。
3. 查看 team agents、健康 JSON 与服务日志。

浏览器路线：

```text
http://127.0.0.1:9898/health
```

引导用户观察：

- `status` 是 `ok` 还是 `degraded`
- planner / executor 模型配置是否存在
- executor candidate queue 与 active probe 结果
- goal mode 与 skill evolution 的摘要字段

讲解重点：

- 这里不是普通 ping，而是带模型配置和 executor 探测信息的运行时健康报告。
- 当 executor 不健康时，job 会给出 environment failure，而不是静默失败。

### 场景 B：创建一个可视化 Job

Desktop UI 路线：

1. 点击左侧 `执行`。
2. 选择 `模型路由`。
3. `执行模式` 选择 `task`。
4. 勾选 `异步执行`。
5. 在 `目标` 中输入：

```text
Create a concise demo report at notes/demo-user-guide-result.md. Include three sections: what was requested, what was produced, and how to verify it.
```

6. 点击 `执行`。
7. 在右侧 `运行状态` 中观察：
   - job status
   - event log
   - Job / Timeline / Events 按钮
8. 点击 `Timeline`，查看内置可视化时间线。

浏览器 / API 路线：

```powershell
$body = @{
  goal = "Create a concise demo report at notes/demo-user-guide-result.md. Include three sections: what was requested, what was produced, and how to verify it."
  mode = "task"
  policy = @{ async = $true }
} | ConvertTo-Json -Depth 6

Invoke-WebRequest -UseBasicParsing `
  -Method Post `
  -Headers @{ Authorization = "Bearer dual-agent-local" } `
  -Uri "http://127.0.0.1:9898/v1/jobs" `
  -ContentType "application/json; charset=utf-8" `
  -Body $body |
  Select-Object -ExpandProperty Content
```

用户会得到类似字段：

```json
{
  "job_id": "job_xxx",
  "status": "running",
  "stream_url": "/v1/jobs/job_xxx/stream",
  "events_url": "/v1/jobs/job_xxx/events",
  "timeline_url": "/v1/jobs/job_xxx/timeline"
}
```

复制 `timeline_url`，打开：

```text
http://127.0.0.1:9898/v1/jobs/<job_id>/timeline
```

也可以打开无鉴权浏览器入口：

```text
http://127.0.0.1:9898/jobs/<job_id>/timeline
```

引导用户观察：

- Timeline 顶部的 job 状态、模式、时间
- 事件卡片从 start / planning / execution / verification 推进到 final
- runtime analysis 中的工具、artifact、verification、阻塞原因摘要
- 页面里是否出现最终输出 preview

讲解重点：

- 这不是一次性黑盒响应，而是可回放、可恢复、可解释的 workflow。
- Timeline 与 `/events` / `/stream` 消费同一组持久化事件。

### 场景 C：Jobs Dashboard 看持久化结果

Desktop UI 路线：

1. 点击左侧 `Jobs`。
2. 查看内嵌 Jobs Dashboard。
3. 从列表进入 job detail / timeline / events。
4. 点击顶部 `浏览器打开`，可以把当前 dashboard 放到系统浏览器里查看。

浏览器路线：

```text
http://127.0.0.1:9898/jobs/dashboard
```

引导用户观察：

- 新 job 是否出现在列表中
- 状态是否从 running 变为 completed / failed / blocked
- 是否能从列表跳转到 detail、events、timeline
- 是否能看到 mode、verified、时间与输出摘要

验证结果文件：

```powershell
Get-Content -Path notes/demo-user-guide-result.md -Encoding utf8
```

讲解重点：

- Dashboard 读取的是持久化 job record，不依赖当前浏览器会话。
- 文件型交付会被写入本地工作区，便于用户直接检查结果。

### 场景 D：实时事件流与历史回放

Desktop UI 路线：

1. 在 `执行` 页创建 job 后，右侧 event log 会实时追加事件摘要。
2. 点击 `Events` 可以打开完整事件 JSON。
3. 点击 `Timeline` 可以查看同一组事件的可视化版本。

API 路线：

将 `<job_id>` 替换为刚才创建的 job：

```powershell
Invoke-WebRequest -UseBasicParsing `
  -Headers @{ Authorization = "Bearer dual-agent-local" } `
  -Uri "http://127.0.0.1:9898/v1/jobs/<job_id>/events" |
  Select-Object -ExpandProperty Content
```

引导用户观察：

- 每个事件都有 `seq`
- 事件包含 `type`、`title`、`summary`、`status`、`phase`
- 刷新或断线后可以用 `since_seq` 继续拉取

可选回放：

```text
http://127.0.0.1:9898/v1/jobs/<job_id>/events?since_seq=3
```

讲解重点：

- 前端可以基于 `seq` 做断线续播。
- Timeline、Dashboard、自定义前端都可以复用这套事件契约。

### 场景 E：Goal Mode 持久目标体验

Desktop UI 路线：

1. 点击左侧 `Goals`。
2. 查看内嵌 Goals Dashboard。
3. 对已有 goal 使用 detail / timeline / events / run-next / review 操作入口。
4. 点击顶部 `浏览器打开`，可以在系统浏览器中打开 Goals Dashboard。

API 路线：

创建一个 goal：

```powershell
$body = @{
  goal = "Prepare a small release readiness checklist for this repository."
  tasks = @(
    @{ title = "Check repository health"; description = "Summarize current docs and runtime readiness."; mode = "task" },
    @{ title = "Create a release checklist"; description = "Write a concise checklist artifact."; mode = "task" }
  )
} | ConvertTo-Json -Depth 8

Invoke-WebRequest -UseBasicParsing `
  -Method Post `
  -Headers @{ Authorization = "Bearer dual-agent-local" } `
  -Uri "http://127.0.0.1:9898/v1/goals" `
  -ContentType "application/json; charset=utf-8" `
  -Body $body |
  Select-Object -ExpandProperty Content
```

打开：

```text
http://127.0.0.1:9898/goals/dashboard
```

引导用户观察：

- goal 是否进入 dashboard
- goal 状态、任务数、当前任务
- detail / timeline / events 链接
- run-next / retry / resume / review 等操作入口

运行下一步：

```powershell
Invoke-WebRequest -UseBasicParsing `
  -Method Post `
  -Headers @{ Authorization = "Bearer dual-agent-local" } `
  -Uri "http://127.0.0.1:9898/v1/goals/<goal_id>/run-next" `
  -ContentType "application/json; charset=utf-8" `
  -Body "{}" |
  Select-Object -ExpandProperty Content
```

打开 goal timeline：

```text
http://127.0.0.1:9898/goals/<goal_id>/timeline
```

讲解重点：

- Goal 是比 job 更长生命周期的 durable runtime。
- 每次 `run-next` 会推进一个任务，并把执行 job 关联回 goal。
- 用户可以分轮推进、失败后 retry/resume，最后 review。

### 场景 F：Skill Evolution Ops 治理视图

Desktop UI 路线：

1. 点击左侧 `Skill Ops`。
2. 查看 proposal queue、risk、eligibility、next action 与 accepted history。
3. 点击顶部 `浏览器打开`，可以在系统浏览器打开同一页面。

浏览器路线：

```text
http://127.0.0.1:9898/skill-evolution/ops
```

引导用户观察：

- proposal queue 是否有待处理项目
- risk tier、dynamic risk、eligibility、next action
- accepted history 与 rollback guide
- auto-audit / auto-validate / auto-accept 是否被 config、risk、readiness 阻断

讲解重点：

- 这里展示的是“自动化为什么能继续或不能继续”的运营解释。
- 当前默认策略保持保守：低风险试点可以自动 validate，但不会放宽 auto-accept。

如果当前没有 proposal，也可以用它展示空状态和健康摘要：空队列本身说明当前没有待处理的 skill evolution 项。

## 4. 建议演示顺序和话术

建议 15 分钟 Desktop 路线：

1. 启动 `npm run desktop`。
2. 进入 `模型` 页，展示 `config/config.yml` 与 `.env` 可视化配置。
3. 点击 `保存并重启`，切到 `健康` 页看 `/health` 与服务日志。
4. 回到 `执行` 页创建异步 job，观察右侧 event log。
5. 点击 `Timeline`，看事件推进、runtime analysis 与最终状态。
6. 点击 `Jobs`，说明所有任务都会持久化。
7. 点击 `Goals`，说明长期目标和 run-next 控制面。
8. 点击 `Skill Ops`，说明 V2 治理边界和默认保守策略。

建议 15 分钟浏览器/API 路线：

1. 先打开 `/health`，确认运行时和模型状态。
2. 打开 `/jobs/dashboard`，说明所有任务都会持久化。
3. 创建异步 job，立刻打开 timeline，看事件实时推进。
4. 回到 dashboard，看 job 完成后的状态和入口。
5. 打开生成的 `notes/demo-user-guide-result.md`，确认文件落盘。
6. 创建 goal，打开 goals dashboard，执行一次 run-next。
7. 打开 `/skill-evolution/ops`，说明 V2 治理边界和默认保守策略。

建议讲解词：

> 这套系统不是只给一个聊天答案，而是把一次 agent 工作拆成可观测、可回放、可恢复的 workflow。新手可以先在 Desktop UI 里配置模型和密钥，再直接执行任务；用户端可以看到任务状态、事件流、timeline、产物、验证结果和后续操作入口。长目标会进入 Goal 控制面，Skill 自进化则进入独立治理视图，默认不会越过风险和验证边界自动接受变更。

## 5. 用户验收清单

体验结束时，用户应能确认：

- 能在浏览器打开 `/jobs/dashboard`、`/goals/dashboard`、`/skill-evolution/ops`、`/health`。
- 能在 Desktop UI 的 `模型` 页看到并编辑 `config/config.yml` 与 `.env` 常用配置。
- 能在 Desktop UI 的 `执行` 页创建 job，并通过 Job / Timeline / Events 按钮查看结果。
- 能在 Desktop UI 中切换 Jobs / Goals / Skill Ops / Health 视图。
- 能通过 `/v1/jobs` 创建一个异步 job，并拿到 `job_id`、`events_url`、`stream_url`、`timeline_url`。
- 能在 timeline 中看到事件推进和最终状态。
- 能在 dashboard 中看到持久化 job 记录。
- 能在本地看到任务生成的结果文件。
- 能创建 goal，并在 goals dashboard 中看到持久化目标与任务状态。
- 能理解 Skill Evolution Ops 里的 blocked / eligible / next action 不是报错，而是治理边界。

## 6. 常见问题

### `/health` 是 degraded

通常是 planner / executor 模型服务、API key、base_url 或本地模型端口未就绪。

Desktop UI 中的处理方式：

1. 打开 `模型` 页，检查 `API URL`、`模型 ID`、`API Key` 引用。
2. 在 `本地配置` 面板确认 `.env` 密钥是否填写。
3. 点击 `保存并重启`。
4. 打开 `健康` 页查看服务日志。

命令行处理方式：

```powershell
npm run doctor
```

### Job 创建后失败

打开对应 job timeline，看 final 事件和 runtime analysis。常见原因包括：

- 模型服务不可用
- executor candidate 全部不健康
- 任务要求写入受限路径
- 搜索 / 抓取配置缺失

### Dashboard 没有新数据

刷新页面，或确认 job 是否已经创建。也可以打开：

```text
http://127.0.0.1:9898/jobs/data
```

### Skill Evolution Ops 为空

空队列是正常状态。它表示当前没有待审 proposal；仍可展示页面结构、过滤入口、健康摘要和默认保守策略。
