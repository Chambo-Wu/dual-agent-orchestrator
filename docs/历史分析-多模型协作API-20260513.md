## 1. 分析目标

本次从 `$open-multi-agent-workflow` 的视角评估项目是否已经满足以下目标：

- 面向普通用户提供“多模型协作 API 接口”
- 让用户通过简单配置即可使用大模型的规划、分解、编排能力
- 让小模型承担执行，以节省 token 成本并增强一定的隐私保护

本次重点不是直接引入 OMA 依赖，而是判断当前仓库在产品定位、架构和代码层面距离这个目标还有多远，以及下一步应如何演进。

## 2. 结论摘要

结论：**项目已经具备“规划模型 + 执行模型 + 本地工具 + API 兼容层 + 初步 team 编排”的雏形，但还不能算已经满足“普通用户简单配置即可使用多模型协作编排能力”的目标。**

更准确地说：

- 它已经是一个**开发者可用的双模型编排原型**
- 它开始具备 **`runTeam()` 风格的自动拆解与协作能力**
- 但距离**面向普通用户的稳定协作 API 产品**仍有明显差距

如果用 OMA 模式来对齐：

- 面向最终用户的目标编排，建议以 **`runTeam()`** 作为主导心智模型
- 面向内部稳定执行、协议适配、工具调用、重试与审计，仍需要 **`runTasks()` 风格的显式阶段控制**

也就是说，这个仓库未来最合理的方向不是“只做一个 planner loop”，而是发展为：

- 外层：目标驱动的 team orchestration
- 内层：显式、可控、可审计的任务阶段流水线

## 4. 为什么说“还没有满足目标”

### 4.1 面向普通用户的“简单配置”仍然不足

当前配置加载仍是手写的简化 YAML 解析器：

- 只支持非常有限的层级结构
- 依赖固定缩进与键名格式
- 没有 schema 校验
- 没有友好的报错
- 没有 profile / 向导 / 校验命令

这意味着系统更像“开发者自己改配置文件”的工具，而不是“普通用户简单填空即可启用”的产品。[src/config.ts](../src/config.ts) [src/config.ts](../src/config.ts)

任务路由配置也采用了几乎同样的自制解析逻辑，说明配置层的脆弱性是系统性问题，而不是单点问题。[src/task-routing.ts](../src/task-routing.ts) [src/task-routing.ts](../src/task-routing.ts)

### 4.2 现有 API 还不是“协作编排接口”，更像兼容代理层

虽然已经有 `/v1/chat/completions`、`/v1/responses`、`/v1/messages`，但服务仍然主要从“最后一条 `user` 消息”推导任务，而不是围绕“目标、计划、子任务、状态、审批、重试、停止”建立一套原生编排资源模型。[README.md](../README.md) [Readme-CN.md](../Readme-CN.md)

这会导致几个问题：

- 外部客户端只能把系统当成“大模型兼容接口”
- 无法稳定查询计划、子任务、执行历史和中间状态
- 无法给普通用户提供“看得懂的编排过程”
- approval / retry / resume / cancel 都缺乏一等抽象

### 4.3 orchestration 逻辑开始变重，但抽象边界还不清晰

`src/orchestrator.ts` 已经承载了大量职责：

- planner / executor 协议衔接
- route policy 决策
- repo 候选评分
- 证据充分性判断
- tool call 闭环
- fallback 执行路径

这说明项目功能在增长，但模块边界还没有完全稳定。长期看，这会拖慢维护和演进速度。[src/orchestrator.ts](../src/orchestrator.ts) [src/orchestrator.ts](../src/orchestrator.ts) [src/orchestrator.ts](../src/orchestrator.ts)

### 4.4 `team` 路径已经出现，但尚未产品化

`runTeam()` 风格的能力已经存在，这是很强的信号；但它仍更像内部试验路径，而不是稳定对外能力：

- agent 定义过于轻量
- 缺少结构化 task/session/job API
- approval 只是 callback，不是持久化状态机
- synthesis 结果没有沉淀为可查询对象

`TeamConfig` 目前也只暴露了很少的运行控制字段，说明 team runtime 还没有被产品化成真正的平台抽象。[src/types.ts](../src/types.ts)

### 4.5 面向“成本优化与隐私保护”的产品表达还不够完整

仓库的目标强调“大模型负责规划、小模型负责执行”，这在架构上已经成立；但真正要把这件事做成产品，还缺这些关键能力：

- 哪些任务必须走 planner、哪些可以纯 executor
- 哪些内容不得离开本地
- 哪些工具结果只允许本地处理
- 如何为用户解释“这一步为什么用了云端模型”
- 如何量化 token 成本节省

目前这些大多还停留在隐含设计里，没有成为明确的策略模型或可解释配置。

## 5. 分层问题清单

### 5.1 产品与 API 层

主要问题：

- 缺少一等的 `job / session / task / artifact / approval` 资源模型
- 对外接口仍偏协议兼容层，而不是协作工作流 API
- 任务状态、计划树、子任务输出无法自然查询
- 没有“先规划后确认再执行”的用户体验

影响：

- 普通用户无法理解系统内部编排过程
- 上层产品难以构建清晰 UI
- supervisor-worker 模式无法稳定外显

### 5.2 编排运行时层

主要问题：

- `src/orchestrator.ts` 责任过重，策略与执行耦合
- `team` 路径与普通 planner loop 并行存在，但没有统一 orchestration contract
- route policy 主要靠关键词匹配，精度和可维护性有限
- 缺少统一的阶段状态机

对应证据：

- 任务类型判断主要依靠 matcher 文本命中。[src/orchestrator.ts](../src/orchestrator.ts)
- 路由策略默认值大量硬编码在 `task-routing.ts` 中。[src/task-routing.ts](../src/task-routing.ts)

### 5.3 配置与可用性层

主要问题：

- 简化 YAML 解析器太脆弱
- 没有配置 schema 和版本化
- 没有启动前自检
- 没有“最小可用配置模板”和多环境 profile

影响：

- 用户容易因缩进/格式问题启动失败
- 迁移到 macOS / Linux / Windows 时配置细节容易继续分叉

### 5.4 安全与治理层

主要问题：

- 缺少按模型、按工具、按路径、按租户的策略模型
- approval 虽存在，但不是持久可追踪的审批对象
- 没有 secret scope / data residency / model permission 机制
- 缺少用户可读审计视图

### 5.5 观测与可解释性层

主要问题：

- 已有 JSONL 日志，但仍偏开发者定位
- 缺少“给普通用户看的计划解释”
- 缺少面向产品的关键指标

建议增加的指标：

- planner 调用次数
- executor 调用次数
- tool 成功率
- 子任务平均时长
- 平均 token 节省估算
- 云端/本地模型流量占比

## 6. 可行性评估

结论：**可行，而且已有较强演进基础。**

原因：

- 双模型边界已经清楚
- 本地工具执行路径已经成立
- API 兼容层已经存在
- `team` 协作雏形已经具备
- 运行日志和稳定性修补已在持续积累

真正的问题不是“方向错了”，而是“当前还停留在系统原型期，尚未完成产品化抽象”。

换句话说，这个项目不是要推倒重来，而是要：

1. 先把配置、状态机、API 资源模型补齐
2. 再把现有 planner loop 与 team runtime 统一到同一套 orchestration contract
3. 最后才考虑是否引入或对齐更标准的 OMA 运行接口

## 7. 结合 open-multi-agent-workflow 的建议

### 7.1 应选择哪种 OMA 模式

建议主模式：**`runTeam()`**

原因：

- 用户目标是“给一个总目标，系统自动规划、分解、编排”
- 这天然对应 team coordinator 模式
- 当前 `src/team.ts` 已有分解、依赖、调度、综合的关键骨架

### 7.2 为什么仍需要 `runTasks()` 思维

尽管对外推荐以 `runTeam()` 为主，但内部仍需要保留明确的阶段流水线：

- 输入标准化
- 任务类型识别
- 策略选择
- planner 规划
- executor 执行
- 工件审计
- synthesis
- approval / retry / stop

这些步骤都更接近 `runTasks()` 的显式 DAG 或阶段控制。否则系统会越来越难排查 Claude Code 这类协议链路问题。

### 7.3 现阶段是否应该立即引入 OMA 依赖

短期建议：**先不急着全量迁移**

更稳妥的做法是：

- 先在本仓库内抽出统一 orchestration seam
- 让 `planner loop` 和 `team mode` 共用相同的 job/task/artifact contract
- 等内部 contract 稳定后，再评估是否以 OMA runtime 替换或包裹现有实现

原因是当前项目的协议兼容、工具闭环和运行补丁已经比较多，直接迁移框架容易把问题从“可见的混乱”变成“框架内的混乱”。

## 8. 阶段性优化改进方案

### Phase 0：产品边界与配置收敛

目标：先让“普通用户简单配置”成立

建议事项：

- 引入正式配置 schema，替代当前简化 YAML 解析
- 增加 `config validate` / `config init` / `doctor` 命令
- 提供 `local-only`、`hybrid`、`cloud-first` 三种预设 profile
- 增加模型能力声明：`planner`、`executor`、`local_only`、`supports_tools`
- 明确工作目录、runtime、artifact、log 的跨平台路径规则

完成标志：

- 用户无需理解源码即可完成首次配置
- 错误配置能得到友好且明确的提示

### Phase 1：统一 orchestration contract

目标：把现有两条运行路径收敛到同一个核心抽象

建议事项：

- 引入 `Job`, `Plan`, `TaskRun`, `Artifact`, `Approval` 类型
- 把 `runTask()` 与 `runTeam()` 收敛到统一 runtime contract
- 将 `src/orchestrator.ts` 中的策略判断、执行闭环、证据判定拆分成独立模块
- 给 route policy 加版本和来源标记

完成标志：

- 单步 loop 与 team 协作都能生成统一的执行记录
- 后续 API 与 UI 可以稳定消费这些对象

### Phase 2：对外暴露真正的协作 API

目标：从“兼容 OpenAI/Anthropic 协议”升级为“原生协作工作流 API”

建议新增接口：

- `POST /v1/jobs`
- `GET /v1/jobs/:id`
- `GET /v1/jobs/:id/plan`
- `GET /v1/jobs/:id/tasks`
- `POST /v1/jobs/:id/approve`
- `POST /v1/jobs/:id/retry`
- `POST /v1/jobs/:id/cancel`
- `GET /v1/jobs/:id/artifacts`

保留现有 `/v1/chat/completions` 等接口作为兼容层，但不要让它们继续承担主产品能力。

### Phase 3：安全、治理与观测产品化

目标：让“省 token + 隐私保护”从口号变成可验证能力

建议事项：

- 增加 per-model / per-tool / per-path policy
- 增加本地数据不出站策略
- 增加审批节点持久化
- 增加 token / cost / local-vs-cloud metrics
- 增加用户可读执行摘要与审计报告

完成标志：

- 能清楚解释“哪一步为什么用大模型，哪一步只用本地模型”
- 能证明节省了多少 token

### Phase 4：team runtime 深化与 OMA 接缝

目标：让 `runTeam()` 路径成为主战场

建议事项：

- 为 agent 定义能力、工具权限、模型偏好、上下文预算
- 给 decomposition 增加结构化输出校验
- 支持更稳定的 dependency graph 与阶段性 resume
- 为 `team` 模式增加 job 持久化与回放能力
- 评估以 OMA runtime 包裹或替换内部 team orchestration
