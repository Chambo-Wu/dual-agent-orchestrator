# Phase 4 验证层缺口路线图日志

- 日期：2026-05-27
- 范围：`verification layer` 二次收口
- 状态：第一轮实现已启动，结构化持久化、workflow 中间模型 verifier、`insufficient` 语义已落地

## 1. 当前判断

Phase 4 的 verification layer 主链路已经完成：系统级 verifier 已落地，planner 不再只依赖模型自述成功，workflow 内也已有 `verify` 节点，最终 job 会发出 verification 事件。

但当前实现仍属于“可用的验证层”，还没有完全进入“可组合、可审计、可配置的验证平台”。后续优化重点不应再是补一个单点 verifier，而是把验证结果、验证策略和验证事件变成一等运行时契约。

## 2. 已具备能力

1. 系统级 verifier 已接入主链路
   - 文件存在检查
   - JSON parse 检查
   - artifact presence 检查
   - git diff 摘要
   - URL artifact 占位检查

2. workflow `verify` 节点已可执行
   - 可读取依赖 task 的 artifacts
   - 可根据 verifier 结果决定 task completed / failed
   - 已有 pass / fail 单元测试覆盖

3. 最终 job verification 已进入事件流
   - `system.verification_passed`
   - `system.verification_failed`
   - timeline/runtime analysis 可消费验证信号

4. 模型 verifier 已有基础能力
   - 最终 job verification 可根据 `verifier` agent 配置追加模型判断
   - 但尚未进入 workflow 中间 `verify` 节点

## 3. 当前缺口

### 3.1 task 级 verifier 选择

当前 workflow `verify` 节点直接运行默认 verifier 集合，尚未根据 task 类型、role、constraints、agent registry 或配置选择 verifier。

后续目标：

- 支持 `verify` task 指定 verifier profile
- 支持按 task kind / artifact type 自动选择 verifier
- 支持系统 verifier 与模型 verifier 组合执行
- 保持默认路径仍为 system-first

### 3.2 模型 verifier 未接入中间 verify 节点

最终 job verification 已可追加 `createModelVerifier(...)`，但 workflow runtime 内部的 `verify` 节点仍只使用默认系统 verifier。

后续目标：

- workflow runtime 可解析当前配置中的 `verifier` agent
- `verify` 节点可选择是否追加模型 verifier
- 模型 verifier 只作为系统规则不足时的补充，不替代基础系统检查

### 3.3 VerificationResult 未一等持久化

当前 `VerificationResult` 已有 `status / summary / checks` 类型，但 workflow 中间验证结果主要压缩进 task output，最终 job 验证也主要通过事件摘要暴露。

后续目标：

- taskRun 可保存完整 verification result
- job record 可保存最终 verification result
- API response / events / timeline 读取同一份结构化结果
- summary 只作为展示字段，不作为唯一事实来源

### 3.4 verifier 事件粒度偏粗

当前主要只有 `system.verification_passed` 和 `system.verification_failed` 两类 lifecycle event。

后续目标：

- 增加 verifier check 级事件
- 记录 check name、status、detail、duration、related artifact/task
- timeline 可直接定位失败 check 与关联 artifact
- runtime analysis 可统计常见 verification blocker

### 3.5 默认 verifier 仍偏浅层

当前默认 verifier 更偏存在性和格式检查，对“证据充分性、来源质量、输出是否满足验收标准”的判断仍不足。

后续目标：

- `url_reachable` 从占位检查升级为可配置 reachability / freshness 检查
- 增强 search/fetch artifact 的来源数量、可读性和可信度检查
- 对写文件任务增加目标路径、内容非空、格式和预期输出匹配检查
- 对结构化输出增加 schema/profile 驱动检查

### 3.6 `insufficient` 状态语义未充分使用

`VerificationResult.status` 已包含 `verified / insufficient / failed`，但当前主实现基本只返回 `verified / failed`。

后续目标：

- 区分执行错误与证据不足
- 将 `insufficient` 映射到 retry / replan / blocked 策略
- 让 planner 看到明确的“缺证据”而不是泛化失败

### 3.7 验收标准未显式建模

当前 verifier 主要读取 goal、taskRuns 和 artifacts，没有稳定的 machine-checkable acceptance criteria。

后续目标：

- 在 workflow task constraints 或 outputs 中引入验收字段
- 支持 minimum evidence count、required artifact type、required schema、required source domain 等基础约束
- verifier 根据显式验收标准生成 checks

### 3.8 集成测试仍需补齐

现有测试覆盖了 runtime verify pass/fail 和最终 verification event，但还缺少真实 API 链路下的中间 verify 场景。

后续目标：

- `/v1/jobs` 创建包含 `verify` 节点的 workflow
- `/events` 回放 verification check 事件
- `/stream` 实时输出 verification 事件
- 模型 verifier 参与中间 verify 节点
- `insufficient` 触发 retry/replan 的端到端测试

## 4. 推荐执行顺序

### 第一轮：契约收口

1. 扩展 `VerificationCheck` / `VerificationResult` 结构
   - 当前状态：已推进
   - 已为 `VerificationCheck` 增加可选 `status = passed / failed / insufficient`
   - `VerificationResult.status` 已开始真实返回 `insufficient`

2. 将完整 verification result 持久化到 taskRun / job record
   - 当前状态：已推进
   - workflow `verify` 节点会写入 taskRun `verificationResult`
   - 最终 job verification 会写入 job `verificationResult`

3. API、events、timeline 统一读取结构化验证结果
   - 当前状态：部分完成
   - job record/API payload 已携带结构化结果
   - events/timeline 仍主要消费摘要和状态，后续继续推进 check 级事件

### 第二轮：策略收口

1. 为 `verify` task 增加 verifier profile / constraints
   - 当前状态：已推进
   - 已支持 `constraints.verifier_profile = system / system_and_model`
   - 已支持 `constraints.verifier_agent_id` 指定注册 verifier agent
   - 校验规则会限制这些字段只能用于 `kind = verify` 的 task

2. workflow runtime 支持 task 级 verifier 选择
   - 当前状态：已推进
   - 当前已支持按 agent registry 中的 `verifier` role/id 自动追加模型 verifier
   - 当前已支持显式 `system` profile 禁用模型 verifier
   - 当前已支持显式 `system_and_model + verifier_agent_id` 路由到指定 verifier agent

3. 将模型 verifier 接入中间 `verify` 节点
   - 当前状态：已推进
   - workflow 中间 `verify` 节点在配置存在 verifier agent 时会在 system-first checks 后追加 `model_verifier`

### 第三轮：语义收口

1. 正式使用 `insufficient`
   - 当前状态：已推进
   - artifact 缺失或 artifact preview 全空会返回 `VerificationResult.status = insufficient`

2. 将 insufficient 接入 retry / replan / blocked 决策
   - 当前状态：部分完成
   - workflow `verify` 节点会将 `insufficient` 映射为 `blocked`
   - retry/replan 的更细分流仍待后续增强

3. 引入可机器检查的验收标准
   - 当前状态：未开始

### 第四轮：可观测与测试

1. 增加 check 级 verification events
2. timeline 支持 verifier check 与 artifact 联动
3. 补齐 API / stream / replay 集成测试

## 5. 今日结论

Phase 4 verification layer 可以保持“主链路已完成”的判断，但需要新增一个“验证层二次收口”工作流。

后续优先级应从高到低排列为：

1. 结构化结果持久化
2. task 级 verifier 选择
3. 模型 verifier 接入中间 verify 节点
4. `insufficient` 语义落地
5. check 级事件与 timeline 联动
6. 集成测试补齐
