# 前端恢复状态与 CTA 对接

这份说明面向不使用内置 `/timeline` 页的前端。

目标只有两件事：

1. 正确展示服务重启后的 job 恢复状态
2. 在需要时给用户明确的下一步动作

## 1. 推荐接入顺序

优先使用：

1. `POST /v1/jobs`
2. `GET /v1/jobs/:id/stream`
3. 断线后用 `Last-Event-ID` 或 `since_seq`
4. 页面刷新时补一次 `GET /v1/jobs/:id`

原因：

- `/stream` 能看到实时恢复状态变化
- `/jobs/:id` 适合首屏和兜底刷新
- `/events` 适合补历史，不适合单独承担实时状态

## 2. 你需要消费的字段

### 2.1 顶层 `follow`

当旧 job 已经自动切到新的 resumed job 时，响应会包含：

```json
{
  "follow": {
    "type": "resumed_job",
    "source_job_id": "job_old",
    "job_id": "job_new",
    "job_url": "/v1/jobs/job_new",
    "events_url": "/v1/jobs/job_new/events",
    "stream_url": "/v1/jobs/job_new/stream",
    "timeline_url": "/v1/jobs/job_new/timeline"
  }
}
```

前端建议：

- 如果当前正在看旧 job 的详情页，显示“已切换到恢复后的新 job”
- 如果你支持自动跳转，直接改订阅目标到 `follow.stream_url`

### 2.2 `snapshot.recovery`

恢复状态在 `GET /v1/jobs/:id/events` 的 `snapshot` 和 `stream` 的 `job.snapshot` 中都会出现。

关键字段：

```json
{
  "recovery": {
    "status": "recovered",
    "reason": "service_restart",
    "recoverable": false,
    "resumed_to_job_id": "job_new",
    "auto_resume_status": "queued",
    "auto_resume_queued_at": "2026-05-27T09:00:00.000Z",
    "auto_resume_queue_position": 2,
    "auto_resume_batch_size": 5,
    "auto_resume_attempted_at": null,
    "auto_resume_failed_at": null,
    "auto_resume_failure_message": null
  }
}
```

`auto_resume_status` 语义：

- `queued`: 已进入恢复队列，尚未开始
- `running`: 服务正在自动恢复
- `succeeded`: 已创建新的 resumed job
- `failed`: 自动恢复失败，等待人工操作

## 3. 你需要响应的 SSE 事件

### 3.1 `job.snapshot`

用来刷新恢复状态、CTA、队列位置。

恢复相关事件发生后，服务会再次推送 `job.snapshot`，不要只处理第一次。

### 3.2 `job.redirect`

当旧 job 已有明确的新目标时，服务会发送：

```text
event: job.redirect
data: {"type":"resumed_job","job_id":"job_new","stream_url":"/v1/jobs/job_new/stream"}
```

前端建议：

- 立即提示“正在跟随恢复后的 job”
- 支持自动切换 stream 到新的 `job_id`

### 3.3 `job.recovered`

这是源 job 的恢复说明事件，不代表真正执行还在旧 job 内继续。

如果 `meta.resumed_to_job_id` 有值，说明后续执行已经转移到新 job。

### 3.4 `job.auto_resume_started`

说明这个 job 已从排队进入恢复执行。

适合在 UI 上把状态从“排队中”切到“恢复中”。

### 3.5 `job.resumed`

说明恢复成功，且新 job 已创建。

优先读：

- `event.meta.resumed_to_job_id`

## 4. CTA 规范

所有控制面主响应现在都会带 `actions`。

示例：

```json
{
  "actions": [
    {
      "id": "resume_now",
      "label": "Resume Now",
      "kind": "api",
      "method": "POST",
      "href": "/v1/jobs/job_old/resume",
      "emphasis": "primary"
    },
    {
      "id": "open_job_events",
      "label": "Open Job Events",
      "kind": "link",
      "href": "/v1/jobs/job_old/events",
      "emphasis": "secondary"
    }
  ]
}
```

当前动作语义：

- `resume_now`
  - 自动恢复失败后出现
  - 直接调用 `POST /v1/jobs/:id/resume`
- `open_job_events`
  - 查看旧 job 的恢复上下文
- `open_resumed_timeline`
  - 打开新 job 的内置 timeline
- `open_resumed_job`
  - 打开新 job 的详情接口

前端建议：

- `kind: "api"` 渲染成按钮
- `kind: "link"` 渲染成跳转链接
- `emphasis: "primary"` 用主按钮样式

## 5. 推荐展示文案

### queued

- 标题：`Automatic Resume Queued`
- 文案：`This job is waiting for an automatic resume slot (2 of 5).`

### running

- 标题：`Automatic Resume Running`
- 文案：`The service is actively resuming this interrupted job.`

### succeeded

- 标题：`Following Resumed Job`
- 文案：`Execution moved to a new resumed job.`

### failed

- 标题：`Action Required`
- 文案优先显示 `auto_resume_failure_message`
- 主 CTA：`Resume Now`

## 6. 最小前端状态机

建议把恢复态单独建模：

```ts
type RecoveryUiState =
  | { kind: "none" }
  | { kind: "queued"; position?: number; batchSize?: number }
  | { kind: "running" }
  | { kind: "redirect"; jobId: string; streamUrl: string }
  | { kind: "failed"; message?: string };
```

更新规则：

1. 先看 `job.redirect`
2. 再看 `snapshot.follow`
3. 再看 `snapshot.recovery.auto_resume_status`
4. 再渲染 `actions`

## 7. 配置项

自动恢复并发来自：

```yml
policy:
  auto_resume_concurrency: 3
```

这个值会影响：

- 服务重启后最多同时恢复多少个 interrupted jobs
- 前端看到的 `auto_resume_queue_position`
- 用户等待恢复开始的时间

## 8. 不要这样做

- 不要把 `job.recovered` 理解成“旧 job 会继续产出 executor 事件”
- 不要只在首次 `job.snapshot` 时读取恢复状态
- 不要忽略 `job.redirect`
- 不要把 `actions` 写死在前端代码里
