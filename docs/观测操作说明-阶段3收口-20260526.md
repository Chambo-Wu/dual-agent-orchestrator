# Dual Agent Orchestrator 观测操作说明

- 日期：2026-05-26
- 适用范围：阶段 3 收口

## 1. 先看什么

出现问题时，建议按下面顺序看：

1. `npm run doctor`
2. `GET /v1/jobs/:id/runtime-profile`
3. `GET /v1/jobs/:id/events`
4. `GET /v1/jobs/:id/timeline`

这个顺序的原则是：

- 先看环境和配置是不是健康
- 再看 job 运行时能力和依赖诊断
- 再看事件流里的真实执行过程
- 最后用 timeline 理解 DAG、replan、artifact 和阶段上下文

## 2. 如何读 `doctor`

`doctor` 适合回答“系统环境是否能正常跑起来”。

重点看：

- `summary`
  - 看通过数、失败数、总检查数
- `checks`
  - 看具体失败的是配置、模型、代理、可写性还是 search provider
- `recommendations`
  - 直接看建议动作，不必先自己猜

适用问题：

- 服务为什么起不来
- 模型配置是否缺失
- 代理是否异常
- workspace / runtime 是否不可写

## 3. 如何读 `runtime-profile`

`/v1/jobs/:id/runtime-profile` 适合回答“这个 job 运行时实际拥有什么能力”。

重点看：

- `platform`
  - 当前 OS、shell、路径风格
- `filesystem`
  - workspace / runtime root
- `network`
  - proxy 模式和健康状态
- `diagnostics_summary`
  - 当前依赖诊断有没有 warning
- `runtime_profile.diagnostics.dependencyChecks`
  - task routing、search provider、proxy 等依赖项的具体状态
- `tools`
  - 哪些工具可用，哪些是 fallback-only

适用问题：

- 为什么 planner / executor 没按预期使用某个工具
- 为什么网络相关任务表现异常
- 为什么搜索能力缺失或退化

## 4. 如何读 `events`

`/v1/jobs/:id/events` 适合回答“这个 job 到底按什么顺序发生了什么”。

重点看：

- `snapshot`
  - 当前 job 摘要、failure summary、replay 信息
- `events[].type`
  - 看 planner、executor、tool、workflow、approval、recovery 事件序列
- `events[].status`
  - 看当前事件是 success、failed、blocked 还是 awaiting_approval
- `events[].meta.failure_category`
  - 快速定位异常类别

适用问题：

- 为什么 job 卡住
- 为什么进入 replan
- 为什么被 blocked
- 为什么恢复后没有继续执行

## 5. 如何读 `timeline`

`/v1/jobs/:id/timeline` 适合回答“从产品视角，这个 job 当前卡在哪条链路上”。

重点看：

- 页头摘要
  - 当前 step、状态、当前 task、failure summary
- Workflow DAG
  - 当前 active workflow 和 superseded workflow 的关系
- Replan History
  - 哪次 replan 替换了哪条 workflow
- lane focus
  - 现在已支持把聚焦状态持久化到 URL，便于分享同一视图
- Runtime Analysis
  - 可直接点击 verification / artifact / tool / blocker 条目，联动筛选并展开对应事件
- URL 状态
  - 现在已支持把 `workflowFocus`、`analysisFilter`、`analysisValue` 持久化到 URL，刷新和分享后可恢复同一定位视图
- 事件卡片
  - 用于对照 DAG 看每个阶段的实际过程

适用问题：

- 哪个 task 是当前瓶颈
- 哪次 replan 触发了替换
- artifact / verifier / task 关系是否合理
- 如何把“统计视图”直接带到“对应事件”和“对应 workflow lane”

## 6. 一句话判断

- `doctor`：看环境和配置
- `runtime-profile`：看能力和依赖
- `events`：看真实过程
- `timeline`：看结构化上下文和前端视角

## 7. 分享建议

如果你要把问题直接发给别人排查，优先分享：

1. 带 `workflowFocus` 的 timeline 链接
2. 带 `analysisFilter` / `analysisValue` 的 timeline 链接
3. 对应 job 的 `/events` 返回

这样对方打开后，能直接落在同一条 workflow lane、同一类异常或同一组事件上。
