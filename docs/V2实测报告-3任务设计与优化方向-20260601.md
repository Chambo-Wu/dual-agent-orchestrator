## 二、归纳总结：实测观察与潜在优化点

### 观察 1：`crossFileConsistency` 仍有精度优化空间

**现象**：`skill_defect` + `patch_body` 组合下，`crossFileConsistency` 返回 `manifest_stable`，但实际上 `patch_body` 会修改 `SKILL.md` 的 Core Procedure，与 `skill.json` 的 description tag 形成隐式关联。当前逻辑仅在 `recommendedAction === "patch_verification"` 时标记 `manifest_verification_only`，在 `diffSummary.scope === "mixed"` 时标记 `needs_audit`，其余一律 `manifest_stable`。

**影响**：`patch_body` 对 `skill_defect` 的变更可能低估跨文件审计需求，auditor 的 `verification_diff_quality_linked` 检查触发不够精确。

**根因**：`buildProposalQualitySummary` 的 crossFileConsistency 判断过于依赖 `recommendedAction`，未考虑 `patch_body` + `skill_defect` 也可能涉及 verification 字段的间接变更。

**涉及文件**：`src/skill-evolver.ts` 第 149-184 行 `buildProposalQualitySummary`。

**收口判断**：近期可优化。建议基于真实 diff / touched fields 判断，而不是仅依赖 `recommendedAction`。

### 观察 2：Reflection-only 路径下 `regression` 分类较难触发

**现象**：在没有 `baselineRecord`（StoredJobRecord）的情况下，`regression` 分类几乎不可能被触发。`buildValidationResultTaxonomy` 中 `regression` 的条件是 `candidateFailedChecks.length > baselineFailedChecks.length`，但当 `baselineRecord` 为 null 时，`baselineFailedChecks` 来自 reflection 的空数组，candidate 的 failed checks 也被设为相同值。

**影响**：V2-B 的验收标准 "validation report 能解释通过、拒绝、无法判断三类结果" 在缺少 baseline job record 时，`regression` 和 `baseline_failed` 更多依赖真实 baseline/replay 证据，而不是 reflection-only 推断。

**根因**：validator 过度依赖 `baselineRecord` 来构建 baseline 证据，reflection-only 路径下 baseline 和 candidate 的 failed checks 被强制对齐。

**涉及文件**：`src/skill-deployment-validator.ts` 第 420-481 行 `buildValidationResultTaxonomy`。

**收口判断**：不作为当前阻塞缺陷。缺少 baseline job record 时保持保守，倾向 `inconclusive` 或阻断自动接受，符合 V2 的可信边界。

### 观察 3：Auto-Accept Eligibility 在无 baseline record 时保持阻断

**现象**：`autoAcceptReady` 在所有无 `baselineRecord` 的场景下都是 `false`，因为 `trueRuntimeReplayReady` 和 `sameInputComparison.readiness === "ready"` 都无法满足。

**影响**：V2-E 的 "低风险 skill 半自动改进试点" 当前只放宽自动 validate，不放宽 auto-accept；如果没有 baseline job record，auto-accept 不会 ready。

**根因**：`low_risk_pilot_skills` 仅放宽 `auto_validate`，auto-accept 的 readiness gate 链路（runtime replay → same-input comparison → execution evidence → binding ready）仍然要求更强证据。

**涉及文件**：`src/skill-deployment-validator.ts` 第 930-940 行 `decision.autoAcceptReady` 计算。

**收口判断**：这是有意保持的保守边界，不建议改成 baseline-free auto-accept readiness。低风险试点仍应只扩大 auto-validate。

### 观察 4：Auditor `manifest_markdown_capability_consistency` 可增加匹配测试

**现象**：当 manifest 新增 `optionalTools`（如 `http_request`）但 markdown 未提及时，检测能命中 `tool:http_request`；但当 manifest 的 `verification.artifactLabels` 新增一个 label 且 markdown 的 Appendix 中有类似但不完全匹配的文本时，`textIncludesNormalized` 的模糊匹配可能产生误判。

**影响**：跨文件一致性的 false positive / false negative 风险，尤其是 artifact label 名称与 markdown 自然语言描述差异较大时。

**根因**：`normalizeSearchText` 做了 aggressive 的 non-alphanumeric 替换，`textIncludesNormalized` 用 `String.includes` 做子串匹配，对语义等价但措辞不同的描述缺乏容错。

**涉及文件**：`src/skill-auditor.ts` 第 221-223 行 `normalizeSearchText`、第 249-251 行 `textIncludesNormalized`。

**收口判断**：近期可补测试和小幅优化。优先增加 artifact label / markdown 描述差异场景，避免误判扩大。

### 观察 5：`decision.reasonCode` 对证据不足场景可更显式

**现象**：`reasonCode` 为 `insufficient_improvement` 的条件是 `!baselineVerified && candidateFailedChecks.length >= baselineFailedChecks.length`。但在 baseline 和 candidate 的 failed checks 都为空时（常见的 reflection-only 场景），这个条件不成立，会 fall through 到 `passed`。

**影响**：某些证据不足的 validation run 可能被表达得过于乐观，进而影响人工理解 validation 可信度。

**根因**：`reasonCode` 的 fallback 逻辑缺少对 "baseline 和 candidate 都无 failed checks" 场景的显式处理。

**涉及文件**：`src/skill-deployment-validator.ts` 第 677-704 行 `reasonCode` 计算。

**收口判断**：作为后续观察项。若实际控制面出现误导性 `passed`，再增加 `insufficient_evidence` 或显式 `inconclusive` 处理。

## 三、轻收口后的优化方向

### 方向 1：收口 `crossFileConsistency` 判断精度

```
当前：patch_verification → manifest_verification_only | mixed → needs_audit | 其他 → manifest_stable

建议：patch_body + skill_defect → 需检查 candidate 是否实际修改了 verification 相关字段
      如果 diff 中包含 verification.* 路径变更 → manifest_verification_only
      如果 diff 中同时包含 body 和 verification 变更 → needs_audit
```

**优先级**：高。直接影响 auditor gate 的触发准确性。

**改动范围**：`src/skill-evolver.ts` 的 `buildProposalQualitySummary` 函数。

### 方向 2：把 Reflection-Only taxonomy 作为观察项

```
当前：baselineRecord = null → baselineFailedChecks = [] → regression 不可触发

可选方向：
  - 从 reflection.evidence.failedCheckNames 推断 baseline failed checks
  - 从 reflection.evidence.missingRequirements 推断 baseline missing requirements
  - 在 baselineRecord 缺失时，仍用 reflection 数据构建 "inferred baseline"
  - 让 regression 分类在 candidate 引入新 failed check 时可触发
```

**优先级**：低。缺少真实 baseline 时维持保守并不阻塞核心链路，避免把推断证据误当成 replay 证据。

**改动范围**：`src/skill-deployment-validator.ts` 的 `validateSkillEvolutionProposal` 函数中 baseline 证据构建逻辑。

### 方向 3：不建立 Baseline-Free Auto-Accept Readiness 路径

```
当前：autoAcceptReady = true 需要 trueRuntimeReplayReady + sameInputComparison.ready + executionEvidence.direct

收口结论：
  - low_risk_pilot_skills 继续只放宽 auto_validate
  - auto-accept 继续要求 runtime replay / same-input comparison / execution evidence
  - 不新增 baseline-free readiness 旁路
  - ops summary 中保持显式 blocked reason
```

**优先级**：不执行。该方向会削弱 V2 的可信边界。

**改动范围**：无代码改动；文档中保留为已评估但不采纳的边界。

### 方向 4：补强 auditor capability matching 测试

```
当前：normalizeSearchText → aggressive 替换 → String.includes 子串匹配

建议：
  - 对 artifact label 和 success signal，先做精确匹配
  - 精确匹配失败后，再做 token-level overlap（Jaccard >= 0.5）
  - 对 intent 和 tool name 保持精确匹配（这些是枚举值）
  - 新增 "approximate_match" 信号到 check detail，供人工审阅参考
```

**优先级**：中。先补测试，再视误判情况决定是否引入 token-level overlap。

**改动范围**：`src/skill-auditor.ts` 的 `textIncludesNormalized` 和 `checkManifestMarkdownCapabilityConsistency` 函数。

### 方向 5：将 `reasonCode` 证据不足场景列为后续观察

```
当前：!baselineVerified && candidateFailedChecks.length >= baselineFailedChecks.length → insufficient_improvement

可选方向：
  - 新增条件：candidateFailedChecks.length === 0 && baselineFailedChecks.length === 0 时，
    不应 fall through 到 passed，而应返回 inconclusive（因为没有足够的 evidence 来判断 improvement）
  - 或者新增 reasonCode: "insufficient_evidence" 专门处理 baseline 和 candidate 都无 failed checks 的场景
```

**优先级**：低到中。当前先观察实际控制面是否出现误导性结果，再决定是否引入新 reasonCode。

**改动范围**：`src/skill-deployment-validator.ts` 的 `reasonCode` 计算逻辑。

## 四、总结矩阵

| 维度 | V2 现状 | 关键差距 | 优化方向 |
|---|---|---|---|
| **Proposal Quality** | 4 类 reflection 均有 fixture 覆盖，quality metadata 输出完整 | `crossFileConsistency` 可基于真实 diff / touched fields 更精确 | 近期收口 |
| **Auditor Gate** | 跨文件一致性、remediation hints、high-risk manual review 均已实现 | capability matching 需要更多 artifact label / markdown 差异测试 | 近期收口 |
| **Validation Taxonomy** | 6 种 result category、11 个 hard gate、stability score 均已覆盖 | reflection-only 路径下 regression/baseline_failed 依赖推断证据 | 后续观察 |
| **Auto-Accept Eligibility** | gate 链路完整，依赖 baseline job record 与 runtime replay 证据 | 无 baseline 时阻断 auto-accept 是有意保守边界 | 不放宽 |
| **Risk Tiering** | coding_like / research_like 分类正确，recovery policy 输出完整 | `reasonCode` 可进一步区分证据不足 | 后续观察 |

**下一步建议**：优先收口 `crossFileConsistency` 精度与 auditor capability matching 测试；`reasonCode` 与 reflection-only taxonomy 作为后续观察项；baseline-free auto-accept readiness 不进入 V2 修复范围。
