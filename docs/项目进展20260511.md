# 项目进展 2026-05-11

## 今日优化修改

1. 工具执行与执行器闭环

- 修复了执行器在原生 `tool_calls` 之后没有继续把工具结果回传给模型的问题，补上了工具调用后的会话闭环。
- 调整了 `write_file` 成功后的收口逻辑，避免文件已经写出后仍继续驱动小模型空跑。
- 进一步把 `read_file`、`list_files` 成功后的流程也改为直接收口，避免把大体积文件内容再次塞回小模型，导致长时间上传或卡住。

2. `shell_command` 稳定性

- 增加了按命令内容智能选择 PowerShell / `cmd` 的执行策略。
- 修复了 PowerShell 启动失败后不能有效回退的问题，避免卡死在 `pwsh.exe` 启动异常上。
- 保留命令输出到 `runtime/command-results`，便于后续审计和复盘。

3. 路径问题修复

- 新增统一路径模块，明确 `PROJECT_ROOT`、`RUNTIME_ROOT`、`WORKSPACE_ROOT`。
- 修复了 `runtime/runtime/command-results` 这类相对路径漂移问题。
- `runtime`、日志目录、`command-results` 目录改为基于项目根目录的绝对路径生成。

4. Cherry Studio 接口兼容

- 修复了 `/v1/messages` 中 `tool_use.input` 的 JSON 容错问题。
- 当上游传入不规范 `arguments` 时，服务端不再直接因为 JSON 解析失败返回 500。
- 当前 Cherry Studio 侧已恢复正常，任务能执行完成，且未再观察到小模型持续循环。

5. 研究任务与 GitHub 结果处理

- 修复了仓库候选解析只识别 `name`、不能稳定识别 `full_name` 的问题。
- 排名与候选提取逻辑已能正确消费 GitHub 搜索结果工件。
- 研究流程补上了工具闭环和证据工件读取后的后续处理。

6. Anthropic / Claude Code 协议侧收口

- 在 `/v1/messages` 的 Anthropic 工具链路上增加了成功 `write_file` 后的提前结束逻辑。
- 目的是在检测到工具结果已经完成时，直接返回普通 assistant 响应，避免继续下发新的 `tool_use`。

7. `tool mode` 循环控制加强

- 在 `src/index.ts` 的 `runToolMode()` 中补充了轮次计数，超过阈值后会移除 tools，强制小模型返回文本响应而不是继续发起 `tool_use`。
- 对 OpenAI / Anthropic 两条 tool-mode 消息归一化逻辑增加了 `tool_result` 内容截断，避免将超长 JSON 原样回灌到执行器上下文。
- 增加了“研究工件已读取”短路检测：当最近一轮成功读取 `runtime/command-results` 下的工件后，服务端会直接要求总结收口，不再继续工具循环。

  - 最新一次现象中，报告甚至未正常产出，说明问题尚未彻底解决。

## 已定位的问题线索

1. 执行器对大体积 `read_file` 结果较敏感

- GitHub 搜索结果 JSON 很大。
- 小模型在读取 `command-results` 下的完整 JSON 后，可能因为上下文过长、严格 JSON 输出要求、或工具结果摘要策略不合理而卡住。

2. Claude Code 链路与 Cherry Studio 链路表现不同

- Cherry Studio 当前表现稳定。
- Claude Code 仍会出现前端持续上传、后台小模型持续运行的问题，说明两条协议链路的收口条件还不完全一致，或仍存在某个未覆盖的循环入口。

3. 研究任务的纠偏逻辑仍可能过强

- 当前系统对 research 任务要求“先读证据、再最终回答”。
- 虽然已经放宽“成功写文件后不要再拉回补证据”，但 Claude Code 场景下仍可能在“读完证据后的小模型总结阶段”卡住。

4. 执行器输出解析仍有脆弱点

- 当小模型没有按预期返回严格 JSON，而是返回空内容、超长内容、或不规范结构时，规划器仍可能进入补救重试。
- 这会表现为界面持续上传、后台 token 持续增长。

## 当前遗留问题

1. Claude Code 研究类任务仍可能持续上传，后台小模型循环。
2. Claude Code 在部分场景下报告不能稳定产出。
3. 执行器在读取大型研究工件后，仍可能卡在总结输出阶段。
4. 研究任务的终止条件还需要进一步细化，尤其是 Claude Code 的 `/v1/messages` 工具协议链路。

## 建议的后续排查方向

1. 给 `read_file` 增加长度限制或自动摘要模式，不再把超长 JSON 全量喂回执行器。
2. 在执行器层增加“单次成功读取工件后直接交给规划器”的更强制短路，而不是依赖小模型再组织 JSON。
3. 对 Claude Code 链路单独增加更细的日志，记录每一轮 `tool_use`、`tool_result`、assistant 收口状态。
4. 为 research 任务增加更明确的最大阶段数和最大证据读取次数，超过即强制结束并返回当前结果。
5. 增加一组针对 `Claude Code + /v1/messages + research + write_file` 的回归测试。

## 结论

今天已经把路径、工具闭环、Cherry Studio 兼容、研究结果解析等几个关键问题修掉，整体稳定性比最初明显提升。当前主要剩余问题集中在 Claude Code 的研究任务链路，尤其是读取大体积证据工件后的收口与终止条件，后续应重点围绕这一段继续压缩上下文和强化短路策略。
