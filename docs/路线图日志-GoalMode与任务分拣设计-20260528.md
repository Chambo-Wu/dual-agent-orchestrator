## 1. 今日结论

当前仓库已经具备较强的执行内核：

- `planner + executor` 主链路稳定
- `workflow plan` 解析、校验、执行与 `replan` 已落地
- `job`、`task run`、`artifact`、`verification`、`resume`、`approve` 已具备平台能力
- 多模型 executor 候选池、主动探测、惰性 warmup 已具备基础

因此，后续不应继续把所有任务都塞进同一条统一执行链路，而应引入顶层“任务分拣层”，在进入执行前先判断任务属于哪种范式，再走不同套路。

今日形成的统一判断是：

1. 系统需要一个顶层 `intent router`
2. 顶层至少区分四类任务：
   - `direct_answer`
   - `research`
   - `goal`
   - `coding`
3. `goal mode` 应作为高于 `job` 的长期目标监督层，而不是直接改写现有 orchestrator 内核
4. `coding mode` 不能继续混入普通 research/task 流程，必须走“读代码 -> 改代码 -> 验证 -> 总结”的工程化套路

## 2. 顶层任务分拣设计

### 2.1 新增顶层 Intent Router

建议新增：

- `src/intent-router.ts`

职责：

- 在所有主入口最前面判断任务意图
- 决定进入 `direct_answer`、`research`、`goal`、`coding` 的哪条执行路径

建议接口：

```ts
type IntentRouteKind =
  | "direct_answer"
  | "research"
  | "goal"
  | "coding";

interface IntentRouteResult {
  kind: IntentRouteKind;
  confidence: number;
  reasons: string[];
  taskType?: string;
  shouldUsePlannerOnly?: boolean;
  shouldCreateJob?: boolean;
}
```

### 2.2 四类任务的判定规则

规则优先，模型补充：

- `goal`
  - 命中 `/goal`、`goal mode`、`拆成任务`、`长期推进`、`large check`、`final review`
- `coding`
  - 命中 `修复`、`改代码`、`加测试`、`重构`、`实现`、`bug`、`报错`、具体文件名/模块名
- `research`
  - 命中 `调研`、`搜索`、`最新`、`案例`、`趋势`、`对比`
- `direct_answer`
  - 解释型、问答型、无需联网、无需改代码、无需拆计划

当规则不确定时，再用轻量 planner 分类器补充判断。

## 3. 四类执行模式

### 3.1 direct_answer

目标：

- planner 可直接回答时，不进入复杂 workflow
- 不默认调用 executor
- 不默认创建重型 job

建议新增：

- `src/direct-answer.ts`

### 3.2 research

目标：

- 继续复用当前 `runTask`
- 检索类首轮可触发 executor lazy warmup
- 保持证据回读、低质量 artifact 拦截、成稿约束

结论：

- research 主链路已较成熟，后续只需作为分拣后的标准路径保留

### 3.3 coding

目标：

- 不再按普通 research 套路走
- 固定为工程化四阶段：
  1. understand
  2. edit
  3. verify
  4. summarize

建议新增：

- `src/coding-runtime.ts`

建议的 coding mode planner 约束：

- 先读代码再改
- 优先本地工具，不默认联网搜索
- 修改后必须验证
- 不能只给建议不落代码

### 3.4 goal

目标：

- 把长期目标拆成 `plan + tasks + checks`
- 每次只推进一个 task
- 每 3 个 task 强制插入一次 `large check`
- 最终执行 `final review`

建议新增：

- `src/goal-types.ts`
- `src/goal-store.ts`
- `src/goal-planner.ts`
- `src/goal-runtime.ts`
- `src/goal-contract.ts`

## 4. Goal Mode v1 设计结论

### 4.1 Goal 不替代 Job

建议的层次关系：

- `goal`
  - 长期目标
  - 持有 `input / plan / tasks / checks / final review`
- `job`
  - goal 的某一次实际执行轮次
- `workflow`
  - 某轮 job 内部的执行计划
- `task run`
  - workflow 内部的最小执行记录

### 4.2 goal.json 作为真源

建议双写：

- 真源：
  - `runtime/goals/<goalId>/goal.json`
- 人类可读镜像：
  - `input.md`
  - `plan.md`
  - `tasks.md`

结论：

- JSON 是状态源
- Markdown 是镜像，不作为唯一事实来源

### 4.3 Goal Mode 的最小状态机

建议：

- `initializing`
- `ready`
- `running`
- `waiting_review`
- `blocked`
- `completed`
- `failed`

### 4.4 Goal Mode 的每轮推进约束

每次推进固定流程：

1. 读取 `goal.json`
2. 找到第一个未完成 task
3. 判断是否需要插入 `large_check`
4. 为当前 task 创建新 job
5. 调用现有 `runTask` / `runTeam`
6. 执行后做验证
7. 回写 task 状态、notes、verificationSummary
8. 更新 markdown 镜像
9. 决定进入下一 task / blocked / waiting_review / completed

## 8. 今日判断的边界

v1 不建议一开始做：

- 多 goal 并发调度器
- 复杂优先级队列
- 跨 goal 共享 memory
- planner 候选池的 goal 级准入逻辑
- 自动 git rollback 执行
- 复杂前端 UI

原因：

- 这些都会显著扩大改动面
- 与“最少代码改动先做出 goal mode v1”的目标不一致
