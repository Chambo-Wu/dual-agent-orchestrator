# 通用多模型协作 API 设计文档
> 文档状态：设计目标，请先阅读 README
>
> 本文描述目标架构、目标接口与阶段性规划，不代表所有内容均已在当前代码中实现。
> 当前实际可用能力、默认端口与对外接口请以 [README](../README.md) 和 [Readme-CN](../Readme-CN.md) 为准。


- 日期：2026-05-22
- 目标：将当前 `dual-agent-orchestrator` 从“可运行的双模型编排器”升级为“通用、跨平台、可恢复、可扩展的多模型协作 API”
- 面向对象：Windows / macOS / Linux 用户；只需配置一个高级模型和一个 worker 模型，即可获得稳定的任务分解与执行能力

---

## 1. 设计目标

### 1.1 核心目标

1. 用户只需最小配置
- 一个高级模型：负责理解需求、规划、验收、总结
- 一个 worker 模型：负责执行工具调用和局部任务

2. 跨平台一致性
- Windows、macOS、Linux 均可运行
- 不要求用户理解平台差异
- 平台差异由 runtime 吸收，而不是交给模型猜测

3. 通用任务支持
- 不仅支持查询、检索
- 还支持文件处理、代码改动、命令执行、数据分析、多步骤任务、多代理协作

4. 一等任务控制面
- 作业可查询、取消、重试、恢复、审批
- 中间工件可追踪、复用、重放

5. 模型与工具解耦
- 模型只消费能力描述和结构化协议
- runtime 决定实际工具、平台、权限、网络策略

### 1.2 非目标

- 不追求成为完整通用浏览器自动化平台
- 不在第一阶段支持任意外部插件生态
- 不在第一阶段解决所有 UI 问题；优先完善 runtime 和 API contract

## 2. 当前系统评估

### 2.1 已具备能力

- planner / executor 分离
- 本地 OpenAI-compatible / Anthropic-compatible 服务入口
- 基本 tool registry
- job / artifact / log 持久化雏形
- team 模式和 decomposition 雏形
- 基本 loop detection、retry、cancel、retry endpoint

### 2.2 当前核心短板

1. 工具能力靠 prompt 暗示，缺少正式 capability negotiation
2. 工具层仍过度依赖 `shell_command`
3. 执行状态过粗，缺少 `partial_success`、`awaiting_approval` 等正式语义
4. artifact 协议较弱，不利于读回、恢复、校验和复用
5. 平台知识散落在 prompt 和工具实现中，未形成 runtime profile
6. 模型配置是“连接信息导向”，不是“能力导向”
7. 缺少独立 verification layer
8. 缺少 durable execution 和 planner / executor step-level resume

## 3. 总体架构

目标架构分四层：

### 3.1 Control Plane

职责：
- 管理 job / plan / planner-executor step / artifact / approval / retry / resume 生命周期
- 提供 HTTP API、状态查询、控制操作
- 对接 UI / CLI / 第三方客户端

核心模块：
- API server
- job store
- artifact store
- approval manager
- session runtime manager

### 3.2 Capability Layer

职责：
- 在运行开始前生成 `RuntimeProfile`
- 明确告知模型当前平台、工具、权限、网络、代理、编码、路径边界
- 统一平台差异

核心模块：
- platform detector
- tool capability registry
- network / proxy probe
- permission profile builder

### 3.3 Execution Layer

职责：
- 运行 planner / worker 协议状态机
- 管理 planner / executor step 执行、部分成功、重规划、人工审批、超时与降级
- 将工具结果转换为稳定工件和结构化状态

核心模块：
- planner runtime
- executor runtime
- verifier runtime
- state machine
- loop / retry / fallback controller

### 3.4 Tool Layer

职责：
- 提供稳定、平台无关的高层工具
- 尽量减少直接 shell 依赖
- 保留 `shell_command` 作为 fallback，而不是主执行路径

核心模块：
- file toolset
- web toolset
- git toolset
- text/data transform toolset
- process/service toolset

## 5. 状态机设计

### 5.1 Job 状态机

```text
queued
  -> running
  -> cancelled

running
  -> awaiting_approval
  -> completed
  -> failed
  -> blocked
  -> cancelled

awaiting_approval
  -> running
  -> blocked
  -> cancelled
```

### 5.2 Step 状态机

```text
pending
  -> running

running
  -> partial_success
  -> awaiting_more_tools
  -> awaiting_approval
  -> needs_replan
  -> completed
  -> failed
  -> blocked

partial_success
  -> completed
  -> needs_replan
  -> awaiting_more_tools

awaiting_more_tools
  -> running
  -> partial_success
  -> failed

needs_replan
  -> pending
  -> blocked
```

### 5.3 关键规则

1. 工具轮次上限不应默认等价于失败
- 若已有有效 artifact / raw_result，应进入 `partial_success`

2. planner 不应直接假设 worker 失败
- 应读取 planner / executor step state 和 verifier 结论

3. 写文件成功不应强制继续搜索
- 若用户目标已满足，可直接进入 verifier 或 final synthesis

4. 工具调用和模型文本结果必须分开建模
- `native_tool_result`
- `model_text_result`
- `verification_result`

## 6. 工具层设计

### 6.1 第一阶段标准工具

#### 文件类
- `read_file`
- `write_file`
- `list_files`
- `stat_file`

#### Web 类
- `web_search`
- `url_fetch`
- `http_request`

#### Git 类
- `git_command`
- `git_diff_file`

#### 文本与结构化处理
- `extract_text`
- `parse_json`
- `parse_csv`
- `summarize_artifact`

#### 进程与服务
- `shell_command`
- `start_service`
- `stop_service`
- `check_port`

### 6.2 工具设计原则

1. 高层能力优先
- 先提供 `web_search`，不要默认让模型自己拼 curl

2. shell 作为 fallback
- `shell_command` 只用于缺少高层工具的场景

3. 每个工具都返回标准结果

```json
{
  "ok": true,
  "summary": "short string",
  "artifacts": [],
  "raw_result": "string",
  "error": "string",
  "verification": {
    "status": "unverified | verified | suspicious",
    "notes": "string"
  }
}
```

4. 工具必须绑定平台约束
- 是否跨平台
- 是否需要 shell
- 是否需要网络
- 是否需要审批

用途：

- 适合轻量、短任务

## 10. Verification Layer 设计

第一阶段 verifier 不必由模型承担全部责任，应优先使用系统规则：

### 10.1 系统级校验

- 文件是否存在
- 输出 JSON 是否符合 schema
- artifact 是否非空
- URL 是否可达
- git diff 是否生成
- 目标路径是否在允许范围内

### 10.2 模型级校验

- 最终报告是否覆盖指定章节
- 结论是否引用 sources
- 代码修复说明是否对应改动

### 10.3 verifier 执行策略

- 系统规则优先
- 模型 verifier 为补充
- 若系统规则已足够，不必再次调用大模型

## 13. 第一阶段验收标准

完成第一阶段后，应满足：

1. Windows / macOS / Linux 用户只需配置 planner + worker，即可完成基础任务
2. 模型不再需要猜当前平台
3. 搜索、抓取、写文件等任务可以产生正式 artifact 并被读回
4. 达到工具轮次上限时，已有成果不会被误判为失败
5. 任一 job 可查询步骤、工件、控制状态
6. 危险操作可以进入审批流
7. 同步接口与 job 控制接口共享同一执行内核

---

## 14. 结论

当前项目最重要的升级方向，不是继续增加单点 prompt 修补，而是把系统抽象提升到：

- 能力协商
- 平台无关工具
- 正式状态机
- 一等控制面
- 可验证工件

这样它才能从“当前这组模型+当前这组工具可以跑”进化到“用户只配两种模型，就能跨平台稳定使用”的通用多模型协作 API。

补充说明（基于 2026-05-25 的当前实现）：`workflow_plan` 主链路已经可以直接执行 `delegate / approval / write / search / fetch / read / extract / verify / synthesize`，其中 `verify` 已接入系统级 verifier；节点级 `retry_policy.fail/skip/fallback/replan` 也已落地，其中 `replan` 已支持 replacement workflow、superseded task history 归档和 `workflow.plan.replanned` / `workflow.task.superseded` 事件；`finish_when` 也已支持 `all_required_tasks_completed / any_of / first_success / manual_approval_resolved` 的执行语义。

后续建议以本设计文档为主线，先收敛 schema 与状态机，再推进 API 和工具层重构。
