## 1. 主骨架总览

当前建议把 Skill 自进化主线拆成 8 个主骨架：

1. `Outcome Capture`
   - 从真实 job / task / verification / artifact / events 中抽取 skill outcome。
2. `Reflection Classifier`
   - 把 outcome 归因为 `discovery / optimization / skill_defect / execution_lapse`。
3. `Proposal Generator`
   - 从 reflection 生成 candidate proposal、candidate skill snapshot、patch summary。
4. `Candidate Materialization`
   - 把 proposal 落成真实 candidate 文件目录，形成可审计、可验证、可回滚的候选版本。
5. `Auditor Gate`
   - 对 candidate 做静态与策略审计，阻止越权与危险 patch。
6. `Deployment Validation`
   - 对 baseline 与 candidate 做验证对比，决定 proposal 是否真的改善。
7. `Decision / Rollback`
   - 负责 accept / reject、live apply、rollback snapshot、decision record。
8. `Observability / Control Plane`
   - 负责 API、events、timeline、dashboard、`/health`、自动链配置和运营视角。

## 2. 当前判断

按当前已修事实同步后，仓库现状看：

- 已打通主链：
  - `Outcome Capture`
  - `Reflection Classifier`
  - `Proposal Generator`
  - `Candidate Materialization`
  - `Auditor Gate`
  - `Deployment Validation`
  - `Decision / Rollback`
  - `Observability / Control Plane`
- 仍属 v1、需要继续做深：
  - `Proposal Generator`
    - 已有 proposal record / candidate snapshot / policy 复用，但候选内容仍偏“最小安全改动”，还不是高质量结构化 rewrite。
  - `Deployment Validation`
    - 已有 candidate-aware validation、risk-tier hard gate、same-input comparison、deterministic isolated manifest replay、replay job/events，并已把 candidate snapshot materialize 成 skill workflow；但还不是完整执行到真实 candidate runtime 的端到端 replay。
- 仍明显缺专项治理：
  - 风险分层自动化策略的运营化说明与可视化
  - 专门的 proposal queue / accepted history / rollback guide 运营视图
  - 全量 skill 的 `SKILL.md` 双结构治理

## 3. 骨架 A：Outcome Capture

### 3.1 定义

目标是把一次 skill 相关运行，稳定抽象为可复用的 `SkillOutcomeSummary`。

### 3.2 当前状态

- 已实现主链。
- 已进入 job response / workflow summary / timeline 语义。
- 已能汇总：
  - `verificationStatus`
  - `failedCheckNames`
  - `missingRequirements`
  - `artifactCount`
  - `taskRunIds`
  - `relatedEventIds`

### 3.3 后续专项路线

1. 收紧 outcome 与 selected skill 的绑定边界。
2. 增强 artifact-to-skill 的归因准确度，减少误挂。
3. 为 replay / validation 补更稳定的 baseline input contract。
4. 为后续风险分层补充 outcome 质量标签。

### 3.4 交付物

- 更明确的 outcome schema 边界
- 更稳定的 baseline replay 输入抽取
- outcome 单测补强

## 4. 骨架 B：Reflection Classifier

### 4.1 定义

目标是把 outcome 归因为可解释的 reflection，并产出 `recommendedAction`。

### 4.2 当前状态

- 已实现四类分类：
  - `discovery`
  - `optimization`
  - `skill_defect`
  - `execution_lapse`
- 已支持 `silent_bypass_signal`
- 已能产出 `append_appendix / patch_body`

### 4.3 后续专项路线

1. 细化 `optimization` 与 `skill_defect` 的边界规则。
2. 将 `silent_bypass_signal` 从布尔信号升级为可解释原因集合。
3. 为不同 skill 类型补差异化分类阈值。
4. 补 classification confidence，便于后续自动化分层。

### 4.4 交付物

- reflection 判定规则文档化
- 分类样本集
- 误判案例回归测试

## 5. 骨架 C：Proposal Generator

### 5.1 定义

目标是基于 reflection 生成 candidate proposal，而不是直接改 live skill。

### 5.2 当前状态

- 已有 proposal record、patch summary、patch text。
- 已能生成 candidate 目录。
- `SKILL.md` 已开始支持结构化候选生成。
- 已开始把 `Reflection-to-Patch` / patch scope / manifest escalation 等策略抽到可被 auditor / validator 复用的 policy 层。
- 但整体仍偏 v1 最小安全改动，不是高质量 skill rewrite。

### 5.3 后续专项路线

1. 把 proposal 生成从“最小改写”升级到“按 reflection 类型分策略生成”。
2. 正式建立 `S_body / S_appendix` 映射策略。
3. 明确 `Discovery / Optimization / SkillDefect / ExecutionLapse` 各自允许改动面。
4. 为 `skill.json` 微调建立白名单字段策略。
5. 再决定是否需要独立 `skill-evolver` 模块或 meta-skill 形态。

### 5.4 交付物

- proposal generation policy
- reflection-to-patch mapping matrix
- 高质量 candidate 生成测试集

## 6. 骨架 D：Candidate Materialization

### 6.1 定义

目标是把 proposal 真正落成 candidate skill 文件，而不是只保留文本提案。

### 6.2 当前状态

- 已有 candidate snapshot 目录。
- 已有 rollback snapshot。
- 已补 candidate `SKILL.md` 结构化落地。
- 缺失 live `SKILL.md` 时也能 scaffold candidate markdown。

### 6.3 后续专项路线

1. 为更多 skill 模板补 candidate materialization 规则。
2. 建立 candidate 与 live 的差异可视化表示。
3. 给 candidate 目录补 changelog / metadata / provenance 信息。
4. 明确 candidate 目录清理、保留和归档策略。

### 6.4 交付物

- candidate file contract
- candidate diff summary
- candidate retention policy

## 7. 骨架 E：Auditor Gate

### 7.1 定义

目标是先审计，再验证，再决定是否可进入下一阶段。

### 7.2 当前状态

- 已有 v1 auditor，并已开始复用 proposal policy 层。
- 已覆盖：
  - `manifest_schema_valid`
  - `skill_markdown_structure_valid`
  - `no_tool_scope_escalation`
  - `no_install_source_escalation`
  - `no_runtime_strategy_escalation`
  - `no_secret_or_task_leakage`
  - `patch_is_non_empty_and_scoped`
  - `reflection_to_patch_consistency`
  - `verification_contract_still_executable`
- 已与 validator 共享部分安全边界：
  - manifest escalation
  - `execution_lapse / append_appendix` patch scope
  - `Core Procedure / Scenario Extensions / Appendix` 段落级 patch 面判断

### 7.3 后续专项路线

1. 为不同 skill 风险等级引入差异化审计策略。
2. 把 markdown 结构检查从“存在 heading”升级到“段落职责约束”。
3. 为 verification 变更增加更细粒度审计项。
4. 补“跨文件一致性”审计，例如 `skill.json` 与 `SKILL.md` 描述漂移。

### 7.4 交付物

- auditor policy matrix
- risk-tier aware audit checks
- 审计失败分类与 remediation 说明

## 8. 骨架 F：Deployment Validation

### 8.1 定义

目标不是看 patch 文本是否“像是更好”，而是验证 candidate 是否真的改善执行结果。

### 8.2 当前状态

- 已有 candidate-aware validation，并已向 isolated replay contract 推进。
- 已比较：
  - baseline / candidate verified
  - artifact count
  - failed checks
  - candidateSelected
- 已补充：
  - baseline selection / input equivalence contract
  - replay provenance / candidate runtime config
  - `silent_bypass` hard gate
  - risk-tier aware validation summary
  - stability / flakiness signal
  - candidate binding readiness
  - execution evidence readiness
  - `sameInputComparison`
  - deterministic `isolated_manifest_replay`
  - baseline / candidate `replayJob`
  - replay-local deterministic event stream
  - candidate runtime `workflow_materialized` 边界
  - candidate runtime workflow execution harness（deterministic replay deps）
- 当前边界：已经不是单纯 heuristic compare，也不再只是 candidate runtime config prepared；manual validate endpoint 已可执行 candidate workflow runtime replay 并在 report 中进入 `stage = executed`。自动 evolution pipeline 也已有默认关闭的 opt-in 开关 `runtime_replay_in_auto_pipeline`，但 auto-accept 放大仍需继续评估。

### 8.3 后续专项路线

1. 继续评估 auto-accept 放大条件：低风险 skill、runtime replay ready、稳定性窗口、dynamic ceiling 全部满足时，是否允许从 opt-in runtime replay 自动进入 accepted。
2. 把 `sameInputComparison` 与 readiness signal 继续汇总到 proposal queue / accepted history / rollback guide 等运营视图。
3. 在 replay event stream 上继续补 task-level step payload。
4. 建立更完整的 non-regression + improvement 双层门槛运营说明。
5. 继续细化 research skill 与 coding skill 的验证策略。

### 8.4 交付物

- isolated replay validator 增强版
- baseline/candidate 对比报告增强版
- validation acceptance contract
- replay readiness / auto-accept readiness 控制面摘要

## 9. 骨架 G：Decision / Rollback

### 9.1 定义

目标是安全完成 accept / reject，并保证 live skill 可回滚。

### 9.2 当前状态

- 已支持 accept / reject API。
- 已支持 apply candidate 到 live skill。
- 已支持 rollback snapshot 保留。
- 已支持 decision record 持久化。

### 9.3 后续专项路线

1. 为 accept 前置更多强约束，尤其是高风险 skill。
2. 把 rollback 从“有快照”推进到“有标准恢复指引”。
3. 增加 accepted history 浏览与回放。
4. 建立 reject taxonomy，沉淀失败原因统计。

### 9.4 交付物

- rollback guide
- accepted history contract
- reject reason taxonomy

## 10. 骨架 H：Observability / Control Plane

### 10.1 定义

目标是让 skill 自进化不是黑箱，而是可查、可审、可运营。

### 10.2 当前状态

- 已有：
  - reflection / proposal / audit / validate / accept / reject API
  - events / stream / timeline / dashboard / `/health`
  - auto pipeline config
  - validation readiness summary 的部分控制面摘要
- 但还没有专门的 evolution 管理视图。

### 10.3 后续专项路线

1. 单独做 proposal queue / accepted history / rollback guide 运营视图。
2. 在 dashboard 上补 proposal funnel。
3. 在 `/health` 上补更细的 evolution 风险与积压指标。
4. 为 auto pipeline 补分层开关和安全说明。
5. 为运维加入 proposal aging、stuck state、频繁失败 skill 告警。

### 10.4 交付物

- evolution ops dashboard
- proposal funnel metrics
- `/health` 扩展字段

## 11. 推荐推进顺序

建议下一步按下面顺序拆专项：

1. `Observability / Control Plane` 专项
   - 把 proposal / validation / auto-accept readiness 变成可运营视图。
2. `Deployment Validation` 专项
   - 在已具备 deterministic isolated manifest replay 的基础上，继续推进真实 candidate runtime replay。
3. `Proposal Generator` 专项
   - 继续提高候选内容质量，这是能力上限。
4. `Auditor Gate` 专项
   - 跟随 proposal 与 validator policy 复用，补更细的策略检查。
5. `SKILL.md` 双结构治理专项
   - 面向全量 skill 做结构补齐。
6. `Automation Risk Tiering` 专项
   - 最后再放大自动 accept。

## 13. 一句话结论

当前 Skill 自进化已经不是“要不要做”的问题，也不再只是 Phase 1-3 的基础链路问题，而是“如何把已打通的 proposal / audit / validation / decision 闭环做深、做稳、做可运营”的问题。

后续最值得拆专项推进的，不是已经打通的骨架，而是：

- `Observability / Control Plane`
- `Deployment Validation`
- `Proposal Generator`
- `SKILL.md` 双结构治理
