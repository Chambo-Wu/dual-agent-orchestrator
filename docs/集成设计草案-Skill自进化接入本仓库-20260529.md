# 集成设计草案：Skill 自进化接入本仓库

- 日期：2026-05-29
- 范围：在现有 `skill registry + skill runtime + verification + timeline` 基础上，引入可观测、可审计、可逐步放开的 `Skill Evolve Workflow`
- 目标：让 Agent 在真实使用中持续改进 skill，但不破坏当前主链稳定性

## 1. 结论

本仓库适合接入 Skill 自进化，但推荐采用分阶段、受控式落地，而不是一次性做“自动修改 skill 并立即生效”。

最契合本仓库的两条思想是：

1. 来自 EmbodiSkill 的“失败归因分层”
   - 区分：
     - `Discovery`
     - `Optimization`
     - `SkillDefect`
     - `ExecutionLapse`
2. 来自 SkillEvolver 的“元技能闭环”
   - 候选 skill 更新
   - 独立审计
   - 部署后验证
   - 通过后再合并

本仓库已经具备以下基础，适合承接该能力：

- 已有 skill 一等对象
  - `src/skill-registry.ts`
  - `src/skill-installer.ts`
  - `src/skill-runtime.ts`
- 已有 skill-aware 选择与观测
  - `candidateSkills`
  - `selectedSkill`
  - install lifecycle events
  - timeline / dashboard
- 已有 skill-aware verification
  - `src/skill-verification.ts`
- 已有 workflow/template 执行主链
  - skill 可以继续复用 `workflow_template`

因此，本方案不建议把“skill 自进化”做成另一个平行系统，而建议把它接成：

```text
job runtime
  -> skill outcome capture
  -> reflection classification
  -> evolve proposal generation
  -> auditor gate
  -> deployment validation
  -> accept / reject
```

## 1.1 当前落地状态（2026-05-29）

本节反映当前代码库中的真实落地结果，不再只是目标设计。

### 已落地

1. Skill-aware outcome / reflection 主链已打通
   - 已有 `skill_outcome`、`skill_reflection` 结构化摘要
   - 已进入 job response / workflow summary / timeline / dashboard
   - 已有 `system.skill_reflection_recorded` 事件语义
2. Skill evolution 手动控制面已打通
   - 已支持 reflection 列表与创建
   - 已支持 proposal 创建、列表、获取
   - 已支持 audit / validate / accept / reject
3. Skill evolution 存储与审计骨架已落地
   - 已有 proposal / audit / validation / decision 持久化目录
   - 已有候选 skill snapshot、rollback snapshot
   - 已有 v1 auditor 与 candidate-aware validation
4. 自动化第一版已接入 job 完成主链
   - `auto_reflect`
   - `auto_propose`
   - `auto_audit`
   - `auto_validate`
   - `auto_accept`
   已可按配置串行推进
5. Observability 与 replay 契约已覆盖到 skill evolution
   - `/events` / `/stream` / timeline / dashboard / `/health`
   - 已有集成测试覆盖 reflection、proposal、audit、validate、accept / reject、auto pipeline

### 已打通但仍属于 v1 启发式

1. `auto_propose` 目前生成的是“最小安全候选变更”
   - 主要是对 `skill.json` 描述、remediation 文本、`SKILL.md` appendix 的最小修改
   - 还不是论文意义上的高质量 skill rewrite
2. deployment validation 目前是 candidate-aware heuristic
   - 会比较 baseline / candidate 的 verification 信号、artifact 数、failed checks
   - 还不是真正把 candidate skill 注入独立 runtime 后做真实 replay 执行
3. auto accept 已能工作，但风险控制仍主要依赖配置开关与现有 auditor
   - 适合作为低风险 skill 的受控自动化
   - 还不适合作为通用默认策略

### 当前剩余缺口

1. proposal 质量仍偏弱
   - 缺少对 `S_body / S_appendix` 的结构化编辑策略
   - 缺少按 `Discovery / Optimization / SkillDefect / ExecutionLapse` 精细化生成不同 patch
2. validation 还不是真 replay
   - 缺少“candidate skill 注入新 Agent / 新 run 再执行”的真实对比链
   - 目前无法充分证明“文本改动真的被用到了”
3. 自动化分层策略还不够细
   - 尚未把不同 skill 按风险等级分层放开
   - 尚未内建“仅低风险 skill 允许 auto_accept”的硬约束
4. `SKILL.md` 双结构约定尚未全面落地
   - 当前并非所有 skill 都具备稳定的 `Core Procedure / Appendix` 分段
   - 这会限制 auto proposal 的质量上限
5. 缺少面向操作者的演进治理视图
   - 还没有专门的“proposal queue / accepted history / rollback guide”管理视图

## 2. 设计原则

### 2.1 先观测，后提案，再验证，最后生效

v1 不应允许：

- 任务失败后直接改写 `skills/`
- 未验证候选 skill 自动进入 builtin 主链
- LLM 自评“我改好了”就放行

v1 应坚持：

1. 先记录真实运行证据
2. 再生成候选 patch
3. 再做 auditor
4. 再做 deployment validation
5. 最后人工或策略批准

### 2.2 自进化只改 skill，不改模型权重

更新对象限定为：

- `SKILL.md`
- `skill.json`
- skill 附属模板
  - `workflow_template`
  - prompt block
  - remediation / verification metadata

### 2.3 优先支持“现有 skill 的 refinement”

v1 只做两类更新：

1. 已存在 skill 的 body/appendix 修订
2. 已存在 skill 的 verification / remediation / template 微调

v1 不优先做：

- 自动从零发明全新 skill
- 自动扩容远程来源 skill
- 自动放大 tool 权限

### 2.4 自进化必须复用现有 observability 主链

不要单独起一套黑盒日志。

所有关键状态都应进入现有：

- job record
- events
- timeline
- dashboard
- `/health`

## 3. 范围与非目标

### 3.1 本方案要解决的问题

1. 某个 skill 被选中了，但执行效果反复不佳
2. 某个 skill 本身没错，但执行时 Agent 经常漏做关键步骤
3. 某个 skill 的验证条件不够强，导致“看起来成功、实际没用”
4. 某个 skill 在新场景下表现出可复用套路，值得沉淀

### 3.2 本方案不解决的问题

1. 通用模型训练或权重更新
2. 远程 skill marketplace
3. skill 签名与供应链体系
4. 多租户 skill 权限隔离平台
5. 通用自我修改引擎

## 4. 总体架构

建议新增一层 `skill evolution control plane`，但仍复用现有 job/workflow 机制。

```text
user task
  -> intent router
  -> skill matcher
  -> selected skill
  -> workflow/task runtime
  -> verification result
  -> skill outcome collector
  -> reflection classifier
  -> evolve proposal
  -> auditor
  -> deployment validator
  -> accepted / rejected proposal
```

建议分成 6 个模块：

1. `skill-outcome`
   - 汇总某次 job 与 skill 相关的运行结果
2. `skill-reflection`
   - 做四类归因
3. `skill-evolver`
   - 生成候选 patch / candidate skill revision
4. `skill-auditor`
   - 做静态与策略检查
5. `skill-deployment-validator`
   - 真实部署验证
6. `skill-evolution-store`
   - 持久化 proposal / audit / validation / decision

## 6. Skill 文件结构升级

## 6.1 目录结构

建议新增约定：

```text
skills/
  find.code_symbol/
    skill.json
    SKILL.md
    changelog.md
  find.official_sources/
    skill.json
    SKILL.md
```

运行时新增：

```text
runtime/
  skill-evolution/
    proposals/
    audits/
    validations/
    accepted/
    rejected/
```

### 6.2 SKILL.md 双结构

建议所有可演化 skill 的 `SKILL.md` 采用固定分段：

```md
# Skill: find.code_symbol

## Core Procedure
...

## Scenario Extensions
...

## Appendix: Execution Notes
...
```

映射关系：

- `Core Procedure`
  - 对应 `S_body`
- `Appendix: Execution Notes`
  - 对应 `S_appendix`

更新约束：

- `Discovery`
  - 可改 `Core Procedure` 或 `Scenario Extensions`
- `Optimization`
  - 可改 `Core Procedure`
- `SkillDefect`
  - 可改 `Core Procedure`
- `ExecutionLapse`
  - 只允许 append 到 `Appendix`

### 6.3 skill.json 的角色

`skill.json` 保持结构化，不鼓励频繁自由修改。

它主要负责：

- intent
- keywords
- tool scope
- execution strategy
- verification contract
- remediation hints

自进化对 `skill.json` 的更新应限制在：

- verification labels
- remediation text
- template references
- activation priority

不应在 v1 自动改：

- `requiredTools`
- `install.source`
- `execution.strategy`

## 9. Reflection 归因工作流

建议新增：

- `src/skill-reflection.ts`

职责：

1. 从 job record + events + verification 生成 `SkillOutcomeSummary`
2. 将 outcome 归因为四类反思类型

### 9.1 归因规则建议

#### Discovery

触发条件：

- job 成功
- verification 通过
- 但出现了新场景关键词或额外人工 note

动作：

- 提议往 `Core Procedure` 或 `Scenario Extensions` 增补适用场景

#### Optimization

触发条件：

- job 成功
- verification 通过
- 但出现明显低效
  - tool retries 多
  - artifact 冗余
  - 绕路后才成功

动作：

- 提议优化 `Core Procedure`

#### SkillDefect

触发条件：

- selected skill 被实际使用
- 相关 verification 不通过
- 且失败不是因为“忘了按 skill 做”

判据建议：

- verification failed / insufficient
- task 与 skill 要求一致
- evidence 表明关键步骤已尝试，但 contract 仍未满足

动作：

- 提议 patch `Core Procedure`
- 必要时微调 verification contract

#### ExecutionLapse

触发条件：

- skill 本身看起来足够
- 失败主要因执行遗漏
  - 应该抓 artifact 没抓
  - 应该 readback 没 readback
  - 应该检查 primary source 没检查

动作：

- 只 append 到 `Appendix`

### 9.2 “静默绕过”检测

这是 SkillEvolver 里最值得保留的一条。

建议把以下情况识别为 `silent_bypass_signal`：

- `selectedSkill` 存在，但最终 workflow/template 没体现 skill 特征
- skill 要求的关键 artifact 没有出现
- skill 相关 verification task 被跳过或形同虚设

它不直接等于 `SkillDefect`，但会作为重要审计风险。

## 10. Evolve Proposal 生成

建议新增：

- `src/skill-evolver.ts`

职责：

1. 根据 `SkillReflectionRecord` 生成候选 patch
2. 输出 candidate skill 目录
3. 产出 proposal record

### 10.1 实现方式

推荐把 evolve 本身做成内置 meta-skill：

```text
skills/meta.skill_evolver/
  skill.json
  SKILL.md
```

执行策略可先选：

- `custom_runtime`

原因：

- 它需要读 skill 文件
- 需要生成 patch
- 需要受控写入 proposal 目录

### 10.2 v1 允许的 patch 范围

允许：

- `SKILL.md`
  - body patch
  - appendix append
- `skill.json`
  - verification labels
  - remediation copy
  - activation priority

不允许：

- 扩大 tool 权限
- 切换 install source
- 新增远程可执行入口

## 11. Auditor 设计

建议新增：

- `src/skill-auditor.ts`

职责：

对 proposal 做机械检查与策略检查。

### 11.1 建议的 v1 审计项

1. `manifest_schema_valid`
   - `skill.json` 仍符合 schema
2. `skill_markdown_structure_valid`
   - `SKILL.md` 含 `Core Procedure` / `Appendix`
3. `no_tool_scope_escalation`
   - `requiredTools` 未扩大
4. `no_install_source_escalation`
   - 未从 `builtin/local_dir` 变为其他来源
5. `no_runtime_strategy_escalation`
   - 未从 `workflow_template` 变成危险 `custom_runtime`
6. `no_secret_or_task_leakage`
   - 没把具体 job 输出、私有路径、用户文本硬编码进 skill
7. `patch_is_non_empty_and_scoped`
   - patch 只影响允许范围
8. `reflection_to_patch_consistency`
   - `ExecutionLapse` 只改 appendix
9. `verification_contract_still_executable`
   - 不会生成当前 runtime 无法验证的要求

### 11.2 审计结果

审计不通过则 proposal 停在：

- `audit_failed`

不会进入 deployment validation。

## 12. Deployment Validation 设计

建议新增：

- `src/skill-deployment-validator.ts`

这是整个方案的关键门槛。

### 12.1 验证目标

不是问模型“这个 patch 看起来好吗”，而是：

- 把 candidate skill 部署到隔离路径
- 用新 job 真跑一次
- 比较 baseline 与 candidate 效果

### 12.2 验证流程

1. 选取一个触发该 skill 的 replay 输入
2. baseline 运行
   - 用当前 skill
3. candidate 运行
   - 用 proposal candidate skill
4. 对比：
   - 是否真的被选中
   - 是否通过 verification
   - artifact 是否更完整
   - failed checks 是否减少
   - 是否出现 silent bypass

### 12.3 v1 判断标准

candidate 通过验证需满足：

1. `candidateSelected === true`
2. 至少不比 baseline 更差
3. 若 baseline 未通过，candidate 应明显改善
4. 不新增高风险审计项

推荐先用“非劣 + 明显改善优先”而不是复杂打分。

## 17. 分阶段落地顺序

### Phase 1：只做 outcome + reflection

目标：

- 不改 skill
- 只产出 reflection 记录

当前状态：

- 已完成
- 已有结构化 `skill_outcome`、`skill_reflection`
- 已进入 events / timeline / dashboard / job response

- `skill outcome summary`
- `skill reflection events`
- timeline 展示

- 某次 skill 失败后，系统能明确标出是 `skill_defect` 还是 `execution_lapse`

### Phase 2：生成 proposal，但不自动运行

目标：

- 可产出 candidate patch
- 不自动生效

当前状态：

- 已完成“手动控制面”版本
- 已支持 proposal store / API / candidate dir
- 当前自动链也可进入 proposal，但 proposal 内容仍偏 v1 最小安全变更

新增：

- proposal store
- proposal API
- candidate dir

验收：

- 可以从一次失败案例稳定生成 patch 提案

### Phase 3：接 auditor

目标：

- 把明显危险或越权的 patch 拦掉

当前状态：

- 已完成 v1 auditor
- 已覆盖 tool scope escalation / install source escalation / runtime strategy escalation / leakage / patch scope 等检查

验收：

- tool scope escalation / strategy escalation / leakage 会被拒绝

### Phase 4：接 deployment validation

目标：

- 不靠自评，靠真实 job 对比

当前状态：

- 已完成 v1 candidate-aware validation
- 当前仍偏启发式 comparison，不是完整独立 replay 执行

验收：

- candidate 真能改善验证结果，才进入 accepted

### Phase 5：有限自动化

目标：

- 对低风险 skill 开启自动 accept

当前状态：

- 自动链第一版已打通
- 已支持 `auto_reflect -> auto_propose -> auto_audit -> auto_validate -> auto_accept`
- 但风险分层放开策略仍未做细，暂不建议默认对高风险 coding skill 全量开启

只建议先开放：

- `find.workspace_files`
- `find.official_sources`

暂不开放：

- `coding` 修改类高风险 skill

## 18. 风险与边界

### 18.1 主要风险

1. skill overfit
   - 为单个案例写死
2. 权限放大
   - 趁 patch 扩大 tool scope
3. 假改进
   - 文本看起来更好，但实际未被使用
4. 验证弱化
   - 通过降低 verification 标准“制造成功”

### 18.2 控制手段

1. proposal 先落隔离目录
2. auditor 拦策略风险
3. deployment validation 做真跑对比
4. 默认人工 accept

## 19. 推荐的 v1 切入 skill

最适合先接入自进化的不是所有 skill，而是：

1. `find.official_sources`
2. `find.workspace_files`

原因：

- 它们偏发现与证据类
- 风险低
- verification contract 清晰
- 更适合积累 `ExecutionLapse` 与 `SkillDefect` 样本

不建议首批接入：

- 直接改代码的高风险 coding skill

## 20. 最终建议

面向本仓库，最稳妥的接入方式不是“让 Agent 自动修改自己的所有技能”，而是：

1. 把每次 skill 使用结果结构化
2. 用 EmbodiSkill 风格归因做 reflection
3. 用 SkillEvolver 风格 proposal + auditor + deployment validation 做闭环
4. 先在低风险 skill 上验证，再逐步放开

一句话概括：

> 本仓库适合接入 Skill 自进化，但应该先把它做成“可审计的提案系统”，再逐步演进为“受控自动改进系统”。
