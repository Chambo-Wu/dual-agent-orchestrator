# Dual Agent Orchestrator 下一阶段执行单页

- 日期：2026-05-26
- 适用范围：`dual-agent-orchestrator`
- 关联基线：[路线图-分阶段持续推进与实施清单-20260526.md](./路线图-分阶段持续推进与实施清单-20260526.md)

## 1. 目标

下一阶段不再重复阶段 1、2 的大面积建设，而是聚焦两件事：

1. 收口阶段 3 剩余产品化与可观测项
2. 启动阶段 4 的最小 team-mode 契约

一句话概括：

先把系统变得更容易看懂、诊断、交付，再把多代理扩展建立在这套稳定契约上。

## 2. 本阶段范围

### 阶段 3剩余项

- timeline 交互抛光
- 运行分析视图
- 观测操作文档
- 事件摘要与异常文案优化

### 阶段 4启动项

- 定义 team-mode 最小可交付 contract
- 明确 `runTask` / `runTeam` 共用语义
- 补 team-mode integration tests 骨架

## 3. 执行顺序

### 第一步：完成阶段 3收口

1. timeline 交互增强
   - lane focus 持久化到 URL
   - replan 前后 plan 对比
   - artifact / verifier / edge 联动

2. 补最小运行分析视图
   - 失败类型统计
   - 常见阻塞点
   - 工具使用分布

3. 补一份观测操作文档
   - 如何读 `doctor`
   - 如何读 `runtime-profile`
   - 如何读 `events`
   - 如何读 `timeline`

4. 抛光事件摘要文案
   - 统一 `title` / `summary`
   - 统一异常分类文案

### 第二步：启动阶段 4

1. 先定义最小 contract
   - `mode: "team"` 请求形态
   - team job 状态语义
   - team 事件流最小集合
   - team 完成判定与失败判定

2. 再打测试骨架
   - team job 创建
   - team job 执行
   - team job 失败
   - team job 恢复

3. 最后推进 runtime 合流
   - `runTask` / `runTeam` 共用 step / artifact / verifier / approval 语义
   - 减少双套状态机

## 4. 本阶段不做

- 不回头把阶段 1、2 重新展开成大规模 backlog
- 不先做 patch 级 replan
- 不先做多角色细粒度模型路由
- 不先做插件化或外部代理接入

这些内容保留到 team-mode 最小路径稳定之后再推进。

## 5. 验收标准

本单页对应的阶段可视为完成，当且仅当：

1. 阶段 3剩余项有明确交付
   - timeline 交互增强已落地
   - 运行分析视图可访问
   - 观测操作文档已成文

2. 阶段 4启动项有明确基线
   - team-mode 最小 contract 已写清
   - team-mode integration tests 已建骨架
   - `runTask` / `runTeam` 统一语义已有明确落点

## 6. 建议产出物

- 代码：
  - timeline 交互增强
  - 运行分析视图
  - team-mode 最小控制面骨架

- 文档：
  - 观测操作文档
  - team-mode 最小 contract 文档

- 测试：
  - team-mode integration tests

## 7. 结论

下一阶段最合理的推进方式，是用最小成本完成阶段 3 收口，并立刻为阶段 4 建立可执行起点。

不要再把注意力分散回已经基本完成的能力面；接下来的关键，是把“可观测、可解释、可交付”的单任务 runtime，稳稳接到“可扩展的 team-mode runtime”上。
