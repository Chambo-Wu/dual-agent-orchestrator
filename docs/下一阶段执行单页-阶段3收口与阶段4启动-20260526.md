# Dual Agent Orchestrator 下一阶段执行单页

- 日期：2026-05-26
- 适用范围：`dual-agent-orchestrator`
- 关联基线：[路线图-分阶段持续推进与实施清单-20260526.md](./路线图-分阶段持续推进与实施清单-20260526.md)
- 最新补充：2026-05-27 先补一轮 verification layer 文档与契约收口，详见 [路线图日志-Phase4验证层缺口-20260527.md](./路线图日志-Phase4验证层缺口-20260527.md)

## 1. 目标

下一阶段不再重复阶段 1、2 的大面积建设，而是聚焦两件事：

1. 先收口 Phase 4 verification layer 的二次契约缺口
2. 收口阶段 3 剩余产品化与可观测项
3. 启动阶段 4 的最小 team-mode 契约

一句话概括：

先把“完成是否可信”变成更结构化、更可审计的契约，再把系统变得更容易看懂、诊断、交付，最后把多代理扩展建立在这套稳定契约上。

## 2. 本阶段范围

### Phase 4 verification layer 二次收口

- 结构化 `VerificationResult` 持久化
- task 级 verifier 选择
- 模型 verifier 接入 workflow 中间 `verify` 节点
- `insufficient` 状态语义落地
- check 级 verifier events

### 阶段 3剩余项

- timeline 交互抛光
- 运行分析视图
- 观测操作文档
- 事件摘要与异常文案优化

### 阶段 4启动项

- 定义 team-mode 最小可交付 contract
- 明确 `runTask` / `runTeam` 共用语义
- 补 team-mode integration tests 骨架

## 3. 执行顺序

### 第一步：完成 verification layer 文档与契约收口

1. 写清结构化 verification result 的落点
   - taskRun
   - job record
   - API response
   - events / timeline

2. 写清 task 级 verifier 选择
   - 默认 system-first
   - 可选 verifier profile
   - 模型 verifier 作为补充路径

3. 写清 `insufficient` 语义
   - 证据不足
   - 执行失败
   - 应 retry / replan / blocked 的分流规则

### 第二步：完成阶段 3收口

1. timeline 交互增强
   - lane focus 持久化到 URL
   - replan 前后 plan 对比
   - artifact / verifier / edge 联动

2. 补最小运行分析视图
   - 失败类型统计
   - 常见阻塞点
   - 工具使用分布

3. 补一份观测操作文档
   - 如何读 `doctor`
   - 如何读 `runtime-profile`
   - 如何读 `events`
   - 如何读 `timeline`

4. 抛光事件摘要文案
   - 统一 `title` / `summary`
   - 统一异常分类文案

### 第三步：启动阶段 4 team-mode

1. 先定义最小 contract
   - `mode: "team"` 请求形态
   - team job 状态语义
   - team 事件流最小集合
   - team 完成判定与失败判定

2. 再打测试骨架
   - team job 创建
   - team job 执行
   - team job 失败
   - team job 恢复

3. 最后推进 runtime 合流
   - `runTask` / `runTeam` 共用 step / artifact / verifier / approval 语义
   - 减少双套状态机

## 4. 本阶段不做

- 不回头把阶段 1、2 重新展开成大规模 backlog
- 不先做 patch 级 replan
- 不先做多角色细粒度模型路由
- 不先做插件化或外部代理接入

这些内容保留到 verification layer 契约和 team-mode 最小路径稳定之后再推进。

## 5. 验收标准

本单页对应的阶段可视为完成，当且仅当：

1. 阶段 3剩余项有明确交付
   - timeline 交互增强已落地
   - 运行分析视图可访问
   - 观测操作文档已成文

2. verification layer 二次收口有明确基线
   - task 级 verifier 选择方案已写清
   - 结构化 verification result 持久化方案已写清
   - `insufficient` 状态语义已写清

3. 阶段 4 team-mode 启动项有明确基线
   - team-mode 最小 contract 已写清
   - team-mode integration tests 已建骨架
   - `runTask` / `runTeam` 统一语义已有明确落点

## 7. 结论

下一阶段最合理的推进方式，是先把 verification layer 的剩余契约缺口写清，再用最小成本完成阶段 3 收口，并继续为阶段 4 team-mode 建立可执行起点。

不要再把注意力分散回已经基本完成的能力面；接下来的关键，是把“可信完成判定、可观测、可解释、可交付”的单任务 runtime，稳稳接到“可扩展的 team-mode runtime”上。
