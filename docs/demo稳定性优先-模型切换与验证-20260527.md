# Demo 稳定性优先：模型切换与验证

- 日期：2026-05-27
- 目标：尽快让 `research/search` 这条 demo 主链路达到可演示状态

## 1. 当前结论

基于两次 demo job 观察与最小直连探针，当前问题更接近：

1. 简单请求可以返回
2. 真实 planner / executor 编排负载下，模型响应稳定性不足
3. 卡点可能出现在 `planner.start`，也可能出现在 `executor.request`
4. 因此当前不应继续把问题归因为单一解析 bug 或单一 `web_search` 工具故障

这意味着第一优先级不是继续加大 prompt 或继续盲跑，而是先把模型组合切到“更稳、更短、更可预测”。

## 2. 推荐原则

1. planner 用更稳定的强模型，负责短控制，不负责长篇生成
2. executor 用延迟更低、tool calling 更稳定的模型
3. research/search 使用专门 executor，限制工具范围
4. 使用 API 型 search provider，避免 demo 期间依赖 HTML 抓取
5. 缩短 history 和 token 上限，减少 orchestrator prompt 体积
6. 减少 replan / retry，让失败尽快暴露，而不是长时间挂住

## 3. 建议配置

仓库已新增可直接套用的样例：

- `config/demo.stability.config.yml`

这份配置做了几件事：

1. planner `max_tokens` 从大上下文收敛到 `4096`
2. planner / executor timeout 收紧到 `45s / 30s`
3. `policy.max_steps` 收敛到 `6`
4. `policy.max_replans` 收敛到 `1`
5. `planner_history_max_entries` 收敛到 `3`
6. `planner_history_preview_chars` 收敛到 `120`
7. `search.provider` 改为 API 型 provider，并关闭 fallback
8. 增加 `researcher` / `writer` executor 分工，降低 research 路径复杂度

## 4. 推荐切换顺序

建议按下面顺序验证，不要一次改太多变量：

1. 先只替换 planner 模型
2. 再替换 research executor 模型
3. 再把 search provider 切到 API 型
4. 最后再决定是否保留 writer agent

如果时间很紧，最小切换集是：

1. 使用 `config/demo.stability.config.yml`
2. 替换 `REPLACE_WITH_*` 占位符
3. 保证 `SEARCH_API_KEY` 可用
4. 先只跑 `research/search` smoke

## 5. 验证标准

建议把 demo smoke 的通过标准压缩成下面四条：

1. `job.started` 后 10 秒内出现新的 planner 或 executor 事件
2. 60 秒内至少产出一个 artifact
3. `/events` 能推进到 `planner.response.parsed` 或 `executor.result`
4. 最终 job 不停留在长期 `running + latest_step pending`

## 6. 验证步骤

1. 用新配置启动服务
2. 先运行 `npm run doctor`
3. 创建一个最小 `research/search` job
4. 观察：
   - `/v1/jobs/:id/runtime-profile`
   - `/v1/jobs/:id/events`
   - `/v1/jobs/:id/timeline`
5. 若 60 秒内仍停在 `planner.start`，优先换 planner
6. 若能过 planner 但停在 `executor.request`，优先换 research executor
7. 若 tool 阶段失败，再看 search provider 和 timeout

## 7. 不建议的做法

当前阶段不建议优先做这些：

1. 继续提高 `max_tokens`
2. 继续放大 planner history
3. 继续依赖 HTML search fallback 做 demo 主路径
4. 在未换模型前，把问题主要归因到 orchestrator 解析层

## 8. 结论

对当前 demo，最现实的路径是：

1. 先切到稳定性优先配置
2. 先验证 `research/search` 单链路
3. 通过后再逐步恢复更高能力配置

这比继续在当前模型组合上盲跑更容易得到可演示结果。
