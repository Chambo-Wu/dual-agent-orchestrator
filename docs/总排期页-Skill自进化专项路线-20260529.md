## 1. 总结论

这 6 个专项不适合并行无序推进。

2026-06-01 同步：Skill 自进化 v1 主链已经落地，当前总排期不再用于安排 outcome/reflection/proposal/auditor/validation 的基础实现，而是用于辅助 `docs/规划-Skill自进化V2-20260601.md` 的 V2 深化与治理收口。

当前最合理的总体顺序是：

1. `SKILL.md` 双结构治理
2. `Proposal Generator`
3. `Deployment Validation`
4. `Observability / Control Plane`
5. `Auditor Gate`
6. `Automation Risk Tiering`

原因很直接：

- `SKILL.md` 双结构治理已经有 inventory/check 设计，是 v2 proposal/auditor 的结构地基。
- `Proposal Generator` 决定候选质量上限。
- `Deployment Validation` 决定闭环可信度下限。
- `Observability / Control Plane` 决定后续推进是否可治理。
- `Auditor Gate` 应跟着 proposal 能力升级而升级。
- `Automation Risk Tiering` 必须放在 proposal / validator / observability 更成熟之后，否则只是把启发式流程更早自动化。

## 2. 专项优先级总表

| 专项 | 优先级 | 当前状态 | 主要价值 | 是否建议立即开工 |
|---|---:|---|---|---|
| SKILL.md 双结构治理 | P0 | inventory/check 与首批 builtin 迁移已完成 | 提升 proposal/auditor 稳定性 | 持续维护 |
| Proposal Generator | P0 | v2 fixtures 首批完成，需扩展真实 skill 场景 | 提高 candidate 质量上限 | 是 |
| Deployment Validation | P0 | runtime replay 已有 manual / opt-in 边界，需评估 auto-accept 放大条件 | 提高闭环可信度 | 是 |
| Observability / Control Plane | P1 | ops summary 已接入，缺独立运营体验与更细 stuck-state | 提高推进与运维效率 | 是 |
| Auditor Gate | P1 | v2 首批规则完成，需 cross-file semantic consistency | 保持安全边界稳定 | 是，但排在 proposal/check 后 |
| Automation Risk Tiering | P2 | 动态风险与 cooldown 已接入，需失败簇和恢复策略 | 决定自动化放开边界 | 不建议最先开工 |

## 3. 依赖关系

## 3.1 核心依赖图

```text
SKILL.md dual-structure governance
          ↓
Proposal Generator ─────→ Auditor Gate
          ↓                    ↓
Candidate Materialization      ↓
          ↓                    ↓
Deployment Validation ─────────┘
          ↓
Observability / Control Plane
          ↓
Automation Risk Tiering
```

## 3.2 具体依赖说明

### Proposal Generator

前置依赖：

- 现有 reflection 主链
- 基础 candidate materialization

后置影响：

- Auditor Gate
- Deployment Validation
- SKILL 双结构治理

### Deployment Validation

前置依赖：

- Proposal Generator
- Candidate Materialization
- Auditor Gate 基础可用

后置影响：

- Accept / Reject 的可信度
- Automation Risk Tiering

### Observability / Control Plane

前置依赖：

- 基础 skill evolution lifecycle 已存在

后置影响：

- 所有专项的运营可见性
- 风险分层落地效率

### Auditor Gate

前置依赖：

- Proposal Generator 至少进入 v2 初期
- Candidate 文件结构较稳定

后置影响：

- Deployment Validation 的输入质量
- Automation Risk Tiering 的安全门

### SKILL.md 双结构治理

前置依赖：

- 基本结构标准已明确

后置影响：

- Proposal Generator 稳定性
- Auditor Gate 结构感知能力

### Automation Risk Tiering

前置依赖：

- Proposal Generator 质量提升
- Deployment Validation 可信度提升
- Observability / Control Plane 可输出 gating 信号

后置影响：

- auto_accept 是否能安全放开

## 4. 推荐开工顺序

## 第 1 批：先做“上限 + 下限”

### 4.1 Proposal Generator

这是第一优先级。

如果 proposal 质量不上去：

- auditor 只能拦很粗的风险
- validation 看到的 candidate 也不够有代表性
- 后续自动化没有意义

### 4.2 Deployment Validation

这也是第一优先级。

如果 validation 不可信：

- accept / reject 的依据不稳
- auto_accept 无法放心开
- proposal 提升再多，也难以形成可信闭环

## 第 2 批：再做“治理与可见性”

### 4.3 Observability / Control Plane

排第三。

原因不是它不重要，而是它应该建立在前两个专项开始成型之后，这样控制面展示的不是“概念状态”，而是真正可运营的状态。

### 4.4 Auditor Gate

排第四。

现在已有 v1 auditor，所以不需要抢在 proposal/validation 前面大改。更合理的方式是：

- 先让 proposal generator 升级
- 再按新 proposal 结构补 auditor

## 第 3 批：做“地基统一”

### 4.5 SKILL.md 双结构治理

排第五，但建议从一开始就小步并行做盘点。

更准确地说：

- 不建议阻塞前两个专项
- 但建议立刻开始 inventory
- 然后按低风险 skill 分批迁移

## 第 4 批：最后做“自动化放大”

### 4.6 Automation Risk Tiering

排最后。

因为它依赖三个前提：

1. proposal 已较稳定
2. validation 已较可信
3. control plane 已能提供明确 gating signal

在这三者不成熟之前，先做 risk tiering 很容易变成“把不稳定流程包装成分层流程”。

## 5. 推荐并行方式

不是完全串行，也不是完全并行。建议采用“主线 + 侧线”模式。

### 主线

1. Proposal Generator
2. Deployment Validation
3. Observability / Control Plane

### 侧线

1. SKILL 双结构治理
   - 先做 inventory
   - 再做低风险 skill 迁移
2. Auditor Gate
   - 跟随 Proposal Generator 的阶段性结果补 check

### 最后收口

1. Automation Risk Tiering

## 6. 推荐阶段排期

## Milestone A：候选质量可控

包含：

- Proposal Generator Phase 1-2
- SKILL 双结构治理 Phase 1-2

目标：

- proposal 不再只是最小安全改写
- `SKILL.md` 结构依赖开始稳定

## Milestone B：验证可信

包含：

- Deployment Validation Phase 1-3
- Auditor Gate Phase 1-2

目标：

- candidate 能被真实注入并参与 replay
- validator 能解释 candidate 为什么通过或失败

## Milestone C：治理可运营

包含：

- Observability / Control Plane Phase 1-3
- Auditor Gate Phase 3

目标：

- proposal queue / accepted history / funnel 可见
- 审计与验证卡点可解释

## Milestone D：受控自动化

包含：

- Automation Risk Tiering Phase 1-4
- Deployment Validation Phase 4-5
- Auditor Gate Phase 4

目标：

- 仅低风险且验证可信的 skill 可 auto_accept
- 高风险 skill 保持人工门槛

## 7. 不建议的开工方式

以下顺序不建议采用：

### 先做 Automation Risk Tiering

问题：

- 会先定义自动化边界，但底层 proposal / validation 还不够稳。

### 先大规模全量迁移 SKILL.md

问题：

- 可能投入很大，但 proposal/validator 还没准备好充分利用这层结构。

### 先重做 Auditor Gate

问题：

- 容易基于旧 proposal 能力过度设计，后面还得重调。

## 8. 推荐资源分配

如果按“小团队连续推进”的节奏，建议：

### 一条主研发线

负责：

- Proposal Generator
- Deployment Validation

### 一条治理支线

负责：

- SKILL 双结构治理
- Observability / Control Plane

### 一条安全收口线

负责：

- Auditor Gate
- Automation Risk Tiering

如果只有单线推进，建议严格按：

1. Proposal Generator
2. Deployment Validation
3. Observability / Control Plane
4. Auditor Gate
5. SKILL 双结构治理
6. Automation Risk Tiering

## 9. 开工建议

如果下一步就要正式排任务，以 `docs/规划-Skill自进化V2-20260601.md` 为准。推荐起手式更新为：

1. 扩展 `Proposal Generator v2` 真实 skill fixtures 与质量分级。
2. 评估 `Deployment Validation v2` runtime replay 下的 auto-accept 放大条件。
3. 设计 `Observability / Ops View v2` stuck-state 与 queue / history 体验。
4. 补 `Auditor Gate v2` cross-file semantic consistency。

这样能最快形成：

- 质量提升主线
- 可信验证主线
- 运营视图支线
- 审计安全支线

## 10. 一句话结论

Skill 自进化这 6 个专项里，当前核心不是“补 v1 主链”，而是：

先把 `SKILL.md` 结构地基、`Proposal Generator` 和 `Deployment Validation` 做深，
再用 `Observability / Control Plane` 和 `Auditor Gate` 把它们治理起来，
最后才进入 `Automation Risk Tiering` 的自动化放大阶段。
